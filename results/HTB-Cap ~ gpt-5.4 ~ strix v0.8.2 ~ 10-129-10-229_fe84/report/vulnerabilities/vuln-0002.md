# Dangerous Linux Capability on Python Interpreter Allows Local Privilege Escalation to Root

**ID:** vuln-0002
**Severity:** HIGH
**Found:** 2026-03-05 19:48:42 UTC
**Target:** 10.129.10.229
**CWE:** CWE-250
**CVSS:** 7.8

## Description

A local privilege escalation vulnerability was confirmed on the target host because the Python interpreter was granted the Linux capability CAP_SETUID. This capability allows a non-root user who can execute the interpreter to change the effective UID of the process to 0 and run commands as root.

Using credentials recovered during the assessment, authenticated access as user nathan was established. Capability inspection confirmed that /usr/bin/python3.8 had CAP_SETUID assigned. Executing a short Python command to invoke os.setuid(0) resulted in immediate root-level code execution and access to protected resources, including the root flag.

This is not a software CVE in Python itself. It is a host hardening failure caused by assigning a powerful privilege-changing capability to a general-purpose interpreter.

## Impact

Any attacker who obtains execution as the low-privilege user nathan can escalate directly to full root privileges on the host. This enables complete compromise of confidentiality, integrity, and availability of the system, including access to all files, credentials, services, and security controls.

In the validated attack chain, a web application weakness exposed credentials, and those credentials were sufficient to gain a local user context. From there, the capability assignment on Python enabled full root compromise with a single command.

## Technical Analysis

The target host was configured with a dangerous file capability on the Python interpreter:

/usr/bin/python3.8 = cap_setuid

Linux file capabilities are intended to grant narrowly scoped privileges to specific binaries. Granting CAP_SETUID to a general-purpose interpreter defeats that model because the interpreter can run arbitrary user-supplied code. A local user can therefore invoke os.setuid(0) and execute shell commands with effective UID 0.

This misconfiguration appears to exist to support the web application's capture functionality, which attempts to call os.setuid(0) before launching tcpdump. However, the design choice exposes the same privilege-escalation primitive to any user who can run the interpreter locally.

Applicability analysis:
Current live lookups against NVD and Ubuntu advisories did not identify a matching product vulnerability for Python 3.8 itself related to this behavior. The issue is environmental and stems from unsafe capability assignment rather than a vendor software defect.

## Proof of Concept

To reproduce:

1. Obtain authenticated access as the local user nathan.
2. Verify the dangerous capability assignment:
   /usr/sbin/getcap /usr/bin/python3.8 /usr/bin/python3 2>/dev/null
3. Confirm output shows:
   /usr/bin/python3.8 = cap_setuid
4. Execute the following command as nathan:
   python3 -c 'import os; os.setuid(0); os.system("id; whoami; cat /root/root.txt")'
5. Observe that the process runs with UID 0 and that protected root-only content is disclosed.

Validated evidence:
The command returned uid=0(root), printed root, and revealed the root flag.

```
import os
import subprocess

def main():
    os.setuid(0)
    subprocess.run(["/bin/sh", "-c", "id; whoami; cat /root/root.txt"], check=True)

if __name__ == "__main__":
    main()
```

## Remediation

1. Remove CAP_SETUID from all general-purpose interpreters, including Python:
   setcap -r /usr/bin/python3.8
2. Review the system for any additional dangerous capabilities assigned to shells, interpreters, or other flexible execution primitives.
3. Redesign packet capture functionality so that privileged actions are mediated through a narrowly scoped helper binary or controlled service rather than a full interpreter.
4. Restrict local access pathways and rotate any credentials exposed during compromise.
5. Perform a full host integrity review because root compromise was demonstrated.
