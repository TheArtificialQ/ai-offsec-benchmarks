# Publicly Accessible Git Metadata Directory Exposes Source Code and Secrets

**ID:** vuln-0001
**Severity:** HIGH
**Found:** 2026-02-28 19:22:28 UTC
**Target:** http://dog.htb
**Endpoint:** /.git/HEAD, /.git/config, /.git/index
**Method:** GET
**CWE:** CWE-552
**CVSS:** 7.5

## Description

Overview:
The web server exposes the Git metadata directory under `/.git/`, allowing unauthenticated remote users to access repository internals over HTTP. This includes Git control files such as `HEAD`, `config`, and index/object data that can be used to reconstruct the application source code.

During validation, repository extraction from the exposed Git directory was successful and disclosed sensitive application information, including configuration data and credentials embedded in source-controlled files.

Severity and CVSS:
This issue is rated High severity due to unauthenticated remote exploitability and high confidentiality impact from full source disclosure and credential leakage.

## Impact

An external attacker can retrieve the application’s source code and sensitive operational data without authentication.

Business impact includes:
Unauthorized disclosure of credentials and internal configuration
Acceleration of follow-on exploitation by revealing routes, logic, and trust boundaries
Potential account compromise through credential reuse across services
Increased likelihood of full system compromise when leaked secrets are chained with other weaknesses

## Technical Analysis

Affected asset:
`http://dog.htb`

Technical details:
The server responds with HTTP 200 for Git control paths under `/.git/`, confirming that repository metadata is web-accessible. Exposure of `.git` allows attackers to enumerate references and object data, then reconstruct committed project files.

Observed behavior included successful access to:
`/.git/HEAD`
`/.git/config`
`/.git/index`

Using standard Git dumping techniques, repository contents were recovered, and sensitive configuration data was obtained from files in the reconstructed codebase. The root cause is deployment misconfiguration: development/version-control artifacts are present in the web-exposed document root and not blocked by server access policy.

Evidence:
HTTP 200 responses from Git metadata endpoints
Successful repository reconstruction from `/.git/`
Sensitive configuration and credential material present in recovered files

## Proof of Concept

Proof of concept:
1. Send unauthenticated HTTP requests to Git metadata endpoints:
   - `GET http://dog.htb/.git/HEAD`
   - `GET http://dog.htb/.git/config`
   - `GET http://dog.htb/.git/index`
2. Confirm `200 OK` responses and Git-identifying content (for example, `ref: refs/heads/...` in `HEAD`).
3. Dump the repository metadata and reconstruct source files using a Git dumping workflow.
4. Review recovered files for sensitive information such as credentials, tokens, and internal configuration.

The script below automates endpoint verification and basic leak confirmation.

```
import re
import requests

BASE = "http://dog.htb"
PATHS = [
    "/.git/HEAD",
    "/.git/config",
    "/.git/index",
]

def fetch(path):
    url = BASE + path
    r = requests.get(url, timeout=10, allow_redirects=False)
    return url, r.status_code, r.text[:400]

def main():
    print("[*] Testing public Git metadata exposure")
    exposed = False

    for p in PATHS:
        url, status, body = fetch(p)
        print(f"{url} -> {status}")
        if status == 200:
            exposed = True
            preview = body.replace("\n", " ")[:200]
            print(f"    preview: {preview}")

    if exposed:
        print("\n[+] Potential .git exposure confirmed.")
        print("[*] Next step (manual): reconstruct repository with a git dumper tool and review for secrets.")
    else:
        print("\n[-] .git endpoints not openly accessible based on tested paths.")

    # Optional heuristic check for DSN-style credentials if HEAD/config are reachable
    try:
        _, s, cfg = fetch("/.git/config")
        if s == 200 and re.search(r"(url\\s*=|\\[remote)", cfg, re.IGNORECASE):
            print("[+] Git config structure detected.")
    except Exception as e:
        print(f"[!] Optional config parsing failed: {e}")

if __name__ == "__main__":
    main()
```

## Remediation

1. Remove `.git` and all version-control artifacts from web-accessible directories in all environments.
2. Enforce explicit web server deny rules for VCS paths and hidden files (for example: `/.git`, `/.svn`, `/.hg`, backup files).
3. Rotate all credentials and secrets that may have been exposed through repository content.
4. Move secrets out of source-controlled files and into a secure secrets management mechanism.
5. Add CI/CD deployment guardrails to block publication of VCS metadata and sensitive files.
6. Perform post-remediation verification by confirming `/.git/*` paths return denied/not found responses and by retesting for repository reconstruction feasibility.
