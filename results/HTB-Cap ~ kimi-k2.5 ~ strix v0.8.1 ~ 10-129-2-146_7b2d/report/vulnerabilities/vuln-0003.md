# Privilege Escalation via Python CAP_SETUID Capability

**ID:** vuln-0003
**Severity:** HIGH
**Found:** 2026-02-22 16:47:47 UTC
**Target:** 10.129.2.146 (cap.htb)
**CWE:** CWE-250
**CVSS:** 7.8

## Description

A critical privilege escalation vulnerability exists on the CAP HTB system due to the Python 3.8 binary having the `cap_setuid` capability enabled (`cap_setuid,cap_net_bind_service+eip`). 

This capability allows any user to change their effective user ID to root (UID 0) programmatically using Python's `os.setuid(0)` function. Combined with the ability to execute arbitrary Python code, this provides a direct path from unprivileged user access to full root privileges.

The vulnerability is particularly dangerous because:
1. It affects the system-wide Python installation at `/usr/bin/python3.8`
2. Any user with shell access can exploit it
3. It provides immediate root access without requiring password authentication
4. No kernel exploits or complex chains are needed

This capability was likely added to support the web application's network capture functionality (which uses tcpdump requiring root), but it inadvertently grants root privileges to all Python processes.

## Impact

Successful exploitation allows any authenticated user to:

- Escalate privileges from standard user (nathan) to root
- Execute arbitrary commands as root
- Access, modify, or delete any files on the system
- Read sensitive files including /root/root.txt
- Install persistent backdoors
- Compromise the entire system

The attack requires only:
1. Valid user credentials (obtained via information disclosure in PCAP files)
2. SSH or shell access to the system
3. Ability to execute Python

This represents complete system compromise with minimal effort.

## Technical Analysis

The root cause is the Linux capability configuration on the Python 3.8 binary:

```
/usr/bin/python3.8 = cap_setuid,cap_net_bind_service+eip
```

The `cap_setuid` capability allows a process to change its real, effective, and saved user IDs to any value, including root (UID 0). When combined with Python's `os.setuid()` function, this provides a trivial privilege escalation path.

The exploit works as follows:
1. Python is executed with elevated capabilities
2. `os.setuid(0)` is called, changing the effective UID to root
3. A new shell is spawned with root privileges
4. All subsequent commands execute as root

This is similar to the SUID bit but more granular - capabilities allow specific privileges without full SUID execution. However, `cap_setuid` is essentially equivalent to SUID root for privilege escalation purposes.

The web application (app.py) actually demonstrates this pattern:
```python
command = f"""python3 -c 'import os; os.setuid(0); os.system("timeout 5 tcpdump...")'"""
```

This shows the developer was aware of the capability and used it for the web app, but the capability is available system-wide to any user.

## Remediation

1. **Remove Dangerous Capability**: Remove the `cap_setuid` capability from Python immediately:
   ```bash
   sudo setcap cap_setuid-eip /usr/bin/python3.8
   ```
   Or remove all capabilities:
   ```bash
   sudo setcap -r /usr/bin/python3.8
   ```

2. **Use Minimal Privileges**: Instead of granting `cap_setuid` to Python, grant only the specific capabilities needed:
   - For tcpdump: `cap_net_raw,cap_net_admin+eip`
   - Do NOT grant `cap_setuid`

3. **Create Dedicated Service User**: Run the web application under a dedicated service account with limited privileges, not as a user that can login via SSH.

4. **Capability Audit**: Audit all binaries with elevated capabilities:
   ```bash
   getcap -r / 2>/dev/null
   ```

5. **Remove Unnecessary Capabilities**: Remove capabilities from any binary that doesn't strictly need them.

6. **Use Sudo Instead**: If elevated privileges are needed for specific commands, use sudo with restricted commands in /etc/sudoers rather than capabilities.

7. **Monitoring**: Monitor for suspicious usage of `os.setuid()` in Python processes via auditd or similar tools.
