# Command Injection via Unsanitized IP Address in Network Capture Feature

**ID:** vuln-0002
**Severity:** CRITICAL
**Found:** 2026-02-22 16:28:49 UTC
**Target:** http://cap.htb
**CWE:** CWE-78
**CVSS:** 9.0

## Description

A critical command injection vulnerability exists in the `/capture` endpoint of the Security Dashboard application. The endpoint captures network traffic using tcpdump and stores it as a PCAP file. The vulnerability is located in lines 105-106 of app.py:

```python
command = f"""python3 -c 'import os; os.setuid(0); os.system("timeout 5 tcpdump -w {path} -i any host {ip}")'"""
os.system(command)
```

The `ip` variable is derived from `request.remote_addr` (client's IP address) and is directly interpolated into a shell command that executes with root privileges (via `os.setuid(0)`). While `remote_addr` is typically server-controlled, the vulnerability can be exploited when:

1. The application is behind a reverse proxy that forwards the client IP via X-Forwarded-For or X-Real-IP headers
2. The proxy does not properly sanitize these headers before passing them to the application
3. An attacker crafts malicious requests with command injection payloads in these headers

The resulting command injection allows arbitrary code execution as the root user, leading to complete system compromise.

## Impact

Successful exploitation allows an attacker to:

- Execute arbitrary system commands as the root user
- Achieve complete system compromise
- Access, modify, or delete any files on the system
- Establish persistent backdoors
- Pivot to internal networks
- Extract sensitive data including all flags and credentials

The vulnerability is particularly severe because:
1. The command executes with root privileges (os.setuid(0))
2. No input validation or sanitization is performed on the IP address
3. The vulnerability is remotely exploitable through HTTP header manipulation

## Technical Analysis

The vulnerability occurs due to unsafe use of `os.system()` with string interpolation of user-influenced input. The problematic code pattern is:

```python
ip = request.remote_addr
command = f"""python3 -c 'import os; os.setuid(0); os.system("timeout 5 tcpdump -w {path} -i any host {ip}")'"""
os.system(command)
```

When Flask is deployed behind a reverse proxy (common with Gunicorn), `request.remote_addr` may be derived from X-Forwarded-For or similar headers. These headers are attacker-controlled and can contain arbitrary strings including shell metacharacters.

The command structure creates a command injection vector:
- The `ip` variable is inserted into a double-quoted shell command
- Shell metacharacters like `;`, `|`, `&`, `$()`, and backticks can break out of the intended command context
- The `os.setuid(0)` call ensures the payload executes with root privileges

For example, if an attacker can inject `8.8.8.8; whoami #` as the IP address, the resulting command becomes:
```bash
python3 -c 'import os; os.setuid(0); os.system("timeout 5 tcpdump -w /path/to/upload/2.pcap -i any host 8.8.8.8; whoami #")'
```

This would execute `whoami` as root in addition to the tcpdump command.

## Proof of Concept

1. Identify that the application uses Gunicorn (observed in Server header)
2. Determine if the application is behind a reverse proxy that forwards X-Forwarded-For headers
3. Craft a malicious request to the `/capture` endpoint with command injection in the X-Forwarded-For header:

```bash
curl -H "X-Forwarded-For: 127.0.0.1; id > /tmp/pwned #" http://cap.htb/capture
```

4. Wait for the capture to complete (5 seconds timeout)
5. Check if the command was executed by attempting to access evidence:
   - Check for output files (e.g., /tmp/pwned)
   - Or observe behavioral changes in the application

Note: Successful exploitation depends on the proxy configuration. If the proxy sanitizes headers or doesn't forward them, the attack may not work directly. Alternative vectors include:
- X-Real-IP header
- Client-IP header
- X-Client-IP header
- CF-Connecting-IP header

## Remediation

1. **Use Parameterized Commands**: Replace `os.system()` with `subprocess.run()` using a list of arguments instead of shell string interpolation:
   ```python
   import subprocess
   subprocess.run([
       "timeout", "5", "tcpdump", 
       "-w", path, "-i", "any", "host", ip
   ], check=True)
   ```

2. **Input Validation**: Validate the IP address format using strict regex or the `ipaddress` module before using it:
   ```python
   import ipaddress
   try:
       ipaddress.ip_address(ip)
   except ValueError:
       return "Invalid IP address", 400
   ```

3. **Remove Privilege Escalation**: Eliminate the `os.setuid(0)` call and configure tcpdump capabilities properly instead of running as root:
   ```bash
   setcap cap_net_raw,cap_net_admin=eip /usr/bin/tcpdump
   ```

4. **Use Werkzeug ProxyFix Carefully**: If using Flask behind a proxy, configure ProxyFix with trusted proxy settings to prevent header spoofing.

5. **Rate Limiting**: The existing rate limiting (10 per minute) helps but is not sufficient to prevent exploitation.

6. **Network Segmentation**: Run the capture functionality in an isolated environment with minimal privileges.
