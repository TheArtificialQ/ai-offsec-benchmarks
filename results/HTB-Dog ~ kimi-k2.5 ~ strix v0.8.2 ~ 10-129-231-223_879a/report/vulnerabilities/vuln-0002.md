# Hardcoded Database Credentials in Configuration File

**ID:** vuln-0002
**Severity:** HIGH
**Found:** 2026-03-03 18:20:38 UTC
**Target:** dog.htb
**Endpoint:** /settings.php
**Method:** GET
**CWE:** CWE-798
**CVSS:** 7.5

## Description

Database connection credentials were found hardcoded in the settings.php configuration file within the web application source code. The credentials (root:BackDropJ2024DS2024) were discovered through the exposed Git repository, allowing potential unauthorized access to the MySQL database.

## Impact

Attackers with access to the source code can obtain database credentials and potentially:
- Access sensitive data stored in the database
- Modify application data
- Escalate privileges within the application
- Pivot to other systems if the database user has excessive permissions

While the MySQL service is only listening on localhost (127.0.0.1), reducing immediate external risk, these credentials could be exploited through:
- Local file inclusion (LFI) vulnerabilities
- Server-side request forgery (SSRF) to localhost
- Command injection vulnerabilities
- Any other vulnerability allowing local command execution

## Technical Analysis

The vulnerability is in settings.php where the database connection string is hardcoded:
$database = 'mysql://root:BackDropJ2024DS2024@127.0.0.1/backdrop';

The root MySQL account should not be used for application database connections. Additionally, credentials should not be stored in plaintext within version-controlled files. The exposure occurred because the .git/ directory was publicly accessible, allowing full source code extraction.

## Proof of Concept

1. Extract source code from exposed .git/ repository
2. Open settings.php file
3. Observe hardcoded database credentials on line 15
4. Credentials found: root:BackDropJ2024DS2024@127.0.0.1/backdrop

```
#!/usr/bin/env python3
"""Proof of Concept for Credential Exposure"""

import requests
import re

TARGET = "http://dog.htb"
SETTINGS_URL = f"{TARGET}/settings.php"

def extract_credentials_from_source():
    """Extract database credentials from settings.php"""
    print("[*] Attempting to access settings.php...")
    
    # Try direct access
    r = requests.get(SETTINGS_URL, timeout=10)
    if r.status_code == 200:
        print("[+] settings.php is accessible!")
        content = r.text
    else:
        print(f"[-] Direct access blocked (status: {r.status_code})")
        print("[*] Trying via .git/ extraction...")
        # Would use git-dumper here in real scenario
        return False
    
    # Extract database credentials
    db_pattern = r"\$database\s*=\s*['\"]([^'\"]+)['\"]"
    match = re.search(db_pattern, content)
    
    if match:
        db_string = match.group(1)
        print(f"\n[+] Database credentials found:")
        print(f"    Connection string: {db_string}")
        
        # Parse components
        if 'mysql://' in db_string:
            parts = db_string.replace('mysql://', '').split('@')
            if len(parts) == 2:
                credentials = parts[0]
                host_db = parts[1]
                print(f"    Credentials: {credentials}")
                print(f"    Host/DB: {host_db}")
        return True
    
    return False

if __name__ == "__main__":
    extract_credentials_from_source()
```

## Remediation

1. Move database credentials to environment variables:
   - Create .env file outside web root
   - Load credentials using getenv() or similar
   
2. Update settings.php:
   $database = getenv('BACKDROP_DATABASE_URL');
   
3. Restrict database user privileges:
   - Create dedicated MySQL user for application
   - Grant only necessary permissions (SELECT, INSERT, UPDATE, DELETE)
   - Do NOT use root account
   
4. Rotate exposed credentials immediately:
   - Change root password
   - Review database access logs for unauthorized access
   
5. Add settings.php to .gitignore to prevent future commits
   
6. Scan repository history for credentials using tools like:
   - git-secrets
   - truffleHog
   - gittyleaks
