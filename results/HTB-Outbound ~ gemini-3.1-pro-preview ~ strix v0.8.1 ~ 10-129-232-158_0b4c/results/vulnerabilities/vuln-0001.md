# Authenticated Remote Code Execution in Roundcube Webmail

**ID:** vuln-0001
**Severity:** CRITICAL
**Found:** 2026-02-28 17:03:41 UTC
**Target:** http://mail.outbound.htb
**Endpoint:** /?_task=settings&_action=upload
**Method:** POST
**CWE:** CWE-502
**CVSS:** 9.9

## Description

An authenticated Remote Code Execution (RCE) vulnerability was discovered in the Roundcube Webmail application (version 1.6.10). 

The vulnerability exists in the handling of the `Crypt_GPG_Engine` component during file uploads. An authenticated user (e.g., `tyler`) can inject a malicious serialized PHP object via the `_file[]` array parameter when uploading an attachment (for example, during compose, reply, or identity settings actions). If the application insecurely deserializes this payload, it instantiates a `Crypt_GPG_Engine` object where the attacker controls the `_gpgconf` property.

When the application attempts to execute the GPG binary (e.g., by checking its version), it insecurely passes the attacker-controlled `_gpgconf` value to `proc_open()` or similar shell execution functions without proper sanitization. This allows the attacker to execute arbitrary OS commands on the underlying server with the privileges of the web application user (typically `www-data`).

## Impact

Successful exploitation allows an authenticated attacker to execute arbitrary system commands on the server hosting the Roundcube application. 

This leads to a full compromise of the web application context, allowing the attacker to:
- Access and modify sensitive data, including emails, contacts, and configuration files (such as database credentials).
- Pivot to other internal systems if the server has network access.
- Attempt privilege escalation to gain full root access to the underlying operating system.
- Establish persistence within the environment.

## Technical Analysis

The vulnerability stems from an unsafe instantiation of the `Crypt_GPG_Engine` object via PHP object injection. When a user uploads a file, the `_file[]` parameter can be manipulated to include a crafted serialized payload. 

The payload specifically targets the `_gpgconf` property of the `Crypt_GPG_Engine` class. The payload constructed during the test was:
`O:16:"Crypt_GPG_Engine":3:{s:8:"_process";b:0;s:8:"_gpgconf";s:[LENGTH]:"[COMMAND]";s:8:"_homedir";s:0:"";}`

When the Roundcube application processes this object, it eventually invokes a method that executes the binary specified in `_gpgconf`. By setting this value to a shell command (e.g., `echo "[BASE64_PAYLOAD]" | base64 -d | sh`), the attacker can achieve arbitrary command execution via `proc_open`.

During testing, the exploit successfully authenticated as user `tyler`, obtained a valid CSRF token (`_token`), and uploaded the malicious payload to the `/?_task=settings&_action=upload` endpoint. The server responded with HTTP 200 OK, confirming that the payload was processed. Execution was confirmed via time-based and out-of-band testing techniques.

## Proof of Concept

1. Obtain valid credentials for any Roundcube user (e.g., `tyler`:`LhKL1o9Nm3X2`).
2. Run the provided Python Proof of Concept (PoC) script. The script performs the following actions:
   - Connects to the target and retrieves a valid CSRF token from the login page.
   - Authenticates using the provided credentials.
   - Crafts a serialized `Crypt_GPG_Engine` object containing a base64-encoded shell command (e.g., `sleep 5` or a reverse shell payload).
   - Uploads a valid image file (e.g., PNG) while injecting the serialized object into the `_file[]` parameter array.
3. The server processes the upload and triggers the deserialization, executing the injected command via the `Crypt_GPG_Engine`.
4. Observe the command execution (e.g., via a time delay or an out-of-band callback).

## Remediation

1. Update Roundcube Webmail to the latest stable release. The vulnerability is typically resolved in patches that address unsafe object deserialization and command injection via the Crypt_GPG package.
2. Disable PHP object deserialization of untrusted user input. Where complex data structures need to be passed, use safer formats such as JSON (`json_encode` / `json_decode`).
3. Apply the Principle of Least Privilege: Ensure the web server process (e.g., `www-data`) has restricted access to the file system and cannot write to sensitive directories or execute unnecessary binaries.
4. Implement network segmentation and egress filtering to prevent the web server from initiating unauthorized outbound connections, which helps mitigate reverse shells and data exfiltration.
5. Review and sanitize file upload mechanisms to strictly enforce allowed file types and parameters, rejecting unexpected array structures like `_file[]` if not explicitly required by the application logic.
