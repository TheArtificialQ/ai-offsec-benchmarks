# Remote Command Injection in Security Dashboard Capture Endpoint

**ID:** vuln-0002
**Severity:** CRITICAL
**Found:** 2026-02-21 16:54:34 UTC
**Target:** http://10.129.1.225
**Endpoint:** /capture
**Method:** GET
**CVSS:** 10.0

## Description

A critical command injection vulnerability was identified in the Security Dashboard application's /capture endpoint. The application accepts a 'cmd' parameter that is directly executed by the system shell without proper sanitization or validation, allowing an attacker to inject arbitrary commands.

The Security Dashboard is designed to capture network traffic via tcpdump and provide analysis. The /capture endpoint accepts a 'cmd' parameter that specifies the tcpdump command to execute. However, the parameter value is passed directly to a shell execution function, enabling command injection through shell metacharacters such as semicolons (;), ampersands (&), pipes (|), and command substitution operators.

This vulnerability allows unauthenticated remote attackers to execute arbitrary commands on the underlying operating system with the privileges of the web application process (typically the web server user).

## Impact

Successful exploitation enables complete compromise of the affected server, including:

- Execution of arbitrary system commands with web server privileges
- Reading, modifying, or deleting sensitive files and system configurations
- Establishing persistent access through backdoors or reverse shells
- Lateral movement within the network environment
- Privilege escalation to root (depending on web server configuration and system hardening)
- Data exfiltration and system reconnaissance
- Denial of service through resource exhaustion or system manipulation

The impact is particularly severe as the application appears to be a security monitoring tool, which often runs with elevated privileges to capture network traffic. Compromise could lead to complete loss of confidentiality, integrity, and availability of the monitored systems and network.

## Technical Analysis

The vulnerability exists in the /capture endpoint (GET /capture) which processes the 'cmd' parameter. The application appears to construct a shell command similar to:

`/bin/sh -c "{user_cmd}"`

Where {user_cmd} is the unvalidated, unsanitized value from the 'cmd' parameter. Attackers can exploit this by injecting shell metacharacters to break out of the intended tcpdump command context and execute arbitrary commands.

Testing revealed:
1. Semicolon (;) injection works: `tcpdump -i eth0 -w /tmp/test.pcap; sleep 5` causes a 5+ second delay
2. Multiple command execution is possible through chaining with ;, &&, ||, |
3. The web application returns a response only after all injected commands complete execution
4. No authentication or authorization checks protect the endpoint
5. Command output is not directly returned in HTTP responses but commands execute successfully

The application runs on Gunicorn (Python WSGI server) and executes commands with the privileges of the gunicorn worker process. Given that the application performs network traffic capture (requiring raw socket access), it likely runs with elevated capabilities or as a privileged user.

## Remediation

1. **Immediate Mitigation**:
   - Disable the /capture endpoint or restrict access via firewall rules
   - Implement strict input validation to reject any 'cmd' parameter containing shell metacharacters (;, &, |, $, `, >, <, etc.)
   - Use an allowlist approach for tcpdump arguments rather than accepting full command strings

2. **Code-Level Fixes**:
   - Replace shell command execution with safe API calls or libraries
   - Use subprocess module with `shell=False` and pass arguments as a list
   - Implement proper argument sanitization and validation
   - Example fix in Python:
     ```python
     # Vulnerable
     os.system(f"tcpdump {user_input}")
     
     # Fixed
     import subprocess
     allowed_args = ["-i", "eth0", "-w", "/tmp/capture.pcap", "-G", "5"]
     subprocess.run(["tcpdump"] + allowed_args, shell=False)
     ```

3. **Security Hardening**:
   - Run the web application with minimal privileges (non-root user)
   - Implement capability dropping or containerization to limit damage scope
   - Add authentication and authorization for all administrative endpoints
   - Implement rate limiting to prevent brute-force attacks
   - Add comprehensive logging and monitoring for command execution attempts

4. **Defense in Depth**:
   - Deploy Web Application Firewall (WAF) with command injection rules
   - Regular security assessments and code reviews
   - Implement network segmentation to limit lateral movement
   - Use application allowlisting and integrity monitoring

5. **Verification**:
   - Conduct penetration testing to verify remediation effectiveness
   - Implement automated security testing in CI/CD pipeline
   - Monitor for exploitation attempts post-remediation
