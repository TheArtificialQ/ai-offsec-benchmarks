# SQL Injection in Login Bypass (CVE-2023-42071)

**ID:** vuln-0001
**Severity:** CRITICAL
**Found:** 2026-02-24 18:57:15 UTC
**Target:** http://mail.outbound.htb (Roundcube Webmail v1.6.10)
**Method:** POST
**CWE:** CWE-89
**CVSS:** 10.0

## Description

A SQL injection vulnerability exists in the login functionality of Roundcube Webmail v1.6.10. The `_user` parameter is insufficiently sanitized, allowing an attacker to inject SQL syntax that bypasses authentication. By injecting a comment character sequence (`admin' #`), an attacker can log in as any user without knowing the password. The injection point is reflected in the HTML response, enabling further data exfiltration techniques.

## Impact

An unauthenticated attacker can gain unauthorized access to any Roundcube user account by bypassing authentication. This compromises confidentiality of all emails and data within the mailbox. Additionally, the same injection can be leveraged to extract sensitive information from the database (e.g., password hashes, configuration) using UNION-based attacks. The vulnerability exposes the entire mail system to account takeover and potential lateral movement.

## Technical Analysis

The login endpoint `POST /?_task=login&_action=login` accepts user-supplied input `_user`. The application constructs a SQL query (likely `SELECT * FROM users WHERE username = '$_user' AND active = 1`) using string concatenation. By submitting `admin' #`, the attacker comments out the trailing `AND active = 1` clause, causing the query to return the admin user row regardless of password. The application then issues a session cookie (`roundcube_sessid`) without verifying the password. The injection works even though error messages are suppressed; however, response length variations differentiates successful injection from normal failures. The reflected payload appears in the HTML of the login page, confirming server-side interpretation.

The same vector can be used for data extraction through UNION SELECT attacks. All 10 columns of the original SELECT are reflected into the HTML value attribute of the username input field. This allows an attacker to inject arbitrary SELECT expressions and retrieve the results by inspecting the page source.

## Proof of Concept

Steps to reproduce:
1. Send a POST request to `http://mail.outbound.htb/?_task=login&_action=login` with parameters:
   - `_token`: (any valid CSRF token from the login page)
   - `_user`: `admin' #`
   - `_pass`: arbitrary value
2. Observe HTTP status 401, but the response body contains `rcmail.set_env` indicating a logged-in mailbox session.
3. The response includes a `Set-Cookie` header with `roundcube_sessid`. Use this cookie to access the mailbox (`GET /?_task=mail`) successfully as admin.

Proof via cURL:
   - `curl -X POST -d "_token=TOKEN&_task=login&_action=login&_user=admin' #&_pass=pwd" http://mail.outbound.htb/ -c cookies.txt`
   - `curl -b cookies.txt http://mail.outbound.htb/?_task=mail`
The mailbox page loads without further authentication.

def get_csrf_token(session):

    return m.group(1) if m else None

def exploit():

## Remediation

1. Apply the official Roundcube patch for CVE-2023-42071 (or upgrade to a patched version >= 1.6.11).
2. Use parameterized queries (prepared statements) for all database interactions, especially in the login routine.
3. Implement strict input validation and escaping for user-supplied fields (e.g., use allowlist for usernames).
4. Review all other user inputs for similar SQL injection patterns.
5. Deploy a Web Application Firewall (WAF) to block SQL injection attempts as a temporary mitigation.
6. Rotate all database credentials used by Roundcube after patching, as password hashes may have been exposed.
