# User Enumeration via Password Reset Endpoint

**ID:** vuln-0005
**Severity:** MEDIUM
**Found:** 2026-02-24 13:32:48 UTC
**Target:** http://dog.htb
**CWE:** CWE-204
**CVSS:** 5.3

## Description

The password reset functionality at /?q=user/password reveals whether a given username or email address exists in the system. By analyzing response messages, an attacker can differentiate between valid and invalid accounts. This was confirmed through testing: submitting 'tiffany' returned an 'Unable to send email' message (indicating valid account), while submitting other usernames like 'admin', 'root', 'user' returned 'invalid' error messages. This vulnerability allows systematic harvesting of valid usernames for subsequent attacks such as password spraying, brute-forcing, or social engineering.

## Impact

An attacker can enumerate valid usernames on the system, significantly reducing the effort required for credential stuffing or brute-force attacks. The revealed usernames can be used for targeted social engineering, password spraying attacks, or as a reconnaissance step for more advanced exploitation. This violates the principle of not revealing account existence to unauthenticated users.

## Technical Analysis

The vulnerability exists in the user password reset form (Form API). The endpoint accepts a 'name' parameter (username or email). The backend performs a user lookup but returns different responses based on whether the user exists:
- Valid user: Response contains error message 'Unable to send email' (email subsystem failure but account exists)
- Invalid user: Response contains error message 'The email address or username is invalid.' or similar

The discrepancy in response content creates a side-channel that reveals account existence. This is a classic observable response discrepancy vulnerability (CWE-204). The issue is compounded by the fact that the site uses predictable usernames (like 'tiffany') and the same password pattern appears in configuration files.

The attack requires no authentication and can be performed at scale against all common usernames and email patterns.

## Proof of Concept

To reproduce:
1. Navigate to http://dog.htb/?q=user/password
2. Enter a username/email (e.g., 'tiffany') in the 'name' field
3. Submit the form
4. Observe the response: an 'Unable to send email' error message appears, indicating the account exists
5. Now submit an invalid username (e.g., 'admin')
6. Observe a different error: 'The email address or username is invalid.'
7. The difference in messages reveals which accounts are valid

Evidence captured: 
- POST request with name=tiffany returned 200 with 'Unable to send email' message
- POST request with name=admin returned 200 with 'invalid' message (from prior automated testing)
- This pattern consistently distinguishes valid from invalid accounts.

## Remediation

1. Implement consistent error messages that do not disclose account existence. The response should be the same whether the username/email is valid or invalid. Example: "If an account with that email or username exists, we have sent a password reset link. Please check your email."
2. Ensure that response times are also consistent (no timing differences) to prevent timing-based enumeration.
3. Consider adding a CAPTCHA or rate limiting on the password reset endpoint to prevent bulk enumeration.
4. Review all other authentication-related endpoints (login, registration) for similar enumeration vulnerabilities.
5. Test the application after changes to verify that enumeration is no longer possible.
