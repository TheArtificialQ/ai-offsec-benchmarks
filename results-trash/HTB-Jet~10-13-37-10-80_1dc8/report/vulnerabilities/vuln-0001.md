# Unauthenticated memory address disclosure via "get gift" in Member manager service (TCP 5555)

**ID:** vuln-0001
**Severity:** HIGH
**Found:** 2026-02-19 18:54:57 UTC
**Target:** 10.13.37.10:5555
**Endpoint:** TCP port 5555 - Member manager "get gift" functionality
**Method:** TCP interactive menu (option 5)
**CVSS:** 7.5

## Description

The custom TCP service running on 10.13.37.10:5555, identified by its banner and menu as "Member manager!", exposes an unauthenticated information disclosure vulnerability in the "get gift" functionality (menu option 5).

Any network client that can connect to TCP port 5555 can:
- Provide an arbitrary name at the initial prompt, and
- Immediately invoke option 5 ("get gift")

to obtain a numeric value that is highly indicative of a raw process memory address (e.g., a 64-bit pointer in the 0x7f.. range), stable within a session and varying across sessions.

This behavior constitutes a strong information leak of internal memory layout. It can be combined with existing or future memory corruption bugs in the same service (e.g., the explicitly labeled "insecure edit" functionality, index-based operations, and size-controlled allocations) to bypass ASLR and other mitigations, significantly lowering the bar for reliable remote code execution against the service.

## Impact

An unauthenticated attacker on the network can reliably obtain pointer-like values from the Member manager process on TCP port 5555.

Concrete impacts include:
- Precise information about the process address space:
  - The leaked value appears as a large 64-bit integer, consistent with a userland pointer (commonly 0x7f************ in hex).
  - The value is stable for the duration of a session and changes between sessions, consistent with a real address (or value derived directly from an address or structure).
- ASLR bypass and exploit reliability:
  - In the presence of any memory corruption primitive (heap overflow, use-after-free, out-of-bounds write) within this service, the infoleak enables an attacker to:
    - Derive the location of heap structures or function pointers, and
    - Construct targeted payloads that achieve reliable code execution despite ASLR.
- Elevated risk from “insecure edit” and index/size handling:
  - The same binary exposes:
    - An "insecure edit" option that allows large user-controlled writes tied to an index parameter.
    - Member creation with user-controlled sizes.
    - Index-based operations (edit, ban) that accept non-trivial values.
  - Even if a corruption bug has not yet been exploited, the existence of a high-quality infoleak combined with these patterns is a strong indicator that RCE would be realistically achievable for a determined attacker.

Business and operational impact:
- Full compromise of the Member manager service process is a plausible outcome when this leak is combined with a memory corruption bug.
- Successful exploitation may lead to:
  - Execution of arbitrary code in the context of the service account.
  - Theft or manipulation of any data handled by the service.
  - Lateral movement to other systems, depending on the privileges and network position of the process.
- The vulnerability is unauthenticated and reachable via a simple, deterministic sequence, increasing the likelihood of discovery and exploitation.

## Technical Analysis

The Member manager service on TCP port 5555 is a custom, menu-driven TCP application. A typical interaction sequence is:

1) On connect, the server displays:
   enter your name:
   (and waits for arbitrary text input)

2) After a name is provided, it presents a menu similar to:
   Member manager!
   1. add
   2. edit
   3. ban
   4. change name
   5. get gift
   6. exit

3) Menu options implement member management operations:
   - add:
     - Prompts "size:" then "username:" and reports "user added!" when successful.
     - Only certain sizes (e.g., 128 and an observed narrow band around the high 400s) are accepted, strongly suggesting direct use of the user-controlled size to perform heap allocations.
   - edit:
     - Offers "1. secure edit" and "2. insecure edit".
     - Both paths take an "index:" and "new username:", but:
       - Secure edit enforces index bounds (invalid index → "no such user!").
       - Insecure edit allows arbitrary indices and accepts large "new username" inputs (thousands of bytes) with no visible crash, indicating potentially unsafe indexing and write behavior.
   - ban:
     - Takes an "index:" and returns "user banned!" or "no such user!".
   - change name:
     - Changes the session's name and accepts long inputs.
   - get gift (vulnerable function):
     - Prints:
       your gift:
       <integer>

Technical characteristics of the leak:
- The integer returned by "get gift" has the following properties:
  - Stable per session:
    - Calling menu option 5 multiple times in the same TCP session always yields the same numeric value.
  - Varies across sessions:
    - Different, fresh connections see different values.
  - Shape and magnitude:
    - Values are large (e.g., around 1.4e14 in decimal, typically converting to addresses of the form 0x7f************), which strongly suggests these are derived directly from pointers or internal addresses (e.g., heap or structure pointers).

- No authentication:
  - The initial "enter your name:" prompt does not enforce any credentials or secret.
  - Immediately after providing any name, the menu is available and "get gift" can be invoked.
  - Therefore, any remote user who can reach TCP port 5555 can trigger the leak.

- Relationship to insecure memory handling:
  - The presence of:
    - An explicitly labeled "insecure edit" path that permits:
      - Arbitrary indices (including negative or large values).
      - Very large "new username" payloads (observed up to at least 16KB).
    - User-controlled allocation sizes for "add" (with only partial validation).
    - Index-based operations for "ban" and "edit".
  - strongly suggests that the underlying implementation uses raw memory management (e.g., malloc/free and manual pointer arithmetic) without comprehensive bounds checking.

  - In such a design:
    - Leaking a pointer (e.g., through "get gift") typically exposes the address of a heap chunk, structure, or function pointer.
    - An attacker can use the leaked pointer as an anchor to calculate the positions of other objects or code segments.
    - This effectively neutralizes ASLR and makes it feasible to build a reliable exploit chain from even a subtle overflow or indexing bug.

- Observed stability:
  - Black-box fuzzing did not induce an immediate crash or visible memory corruption solely via the menu, but this does not mitigate the severity of the leak:
    - The leaked pointer is already a critical primitive on its own.
    - In real-world exploitation, attackers would reverse-engineer the binary or perform offline analysis to identify and exploit latent memory corruption vulnerabilities, using the gift leak to make exploitation robust.

Root cause (conceptual):
- The "get gift" functionality appears to return a value derived directly from process internals, most likely a pointer or an address of a sensitive in-memory object, instead of a non-sensitive, application-level token (such as a random number or a synthetic ID).
- There is no redaction, masking, or translation to an abstract identifier; the service exposes the raw or trivially-transformed pointer value directly to the client.

## Proof of Concept

To reproduce the vulnerability and observe the memory-address leak:

1) From a host that can reach 10.13.37.10:5555, connect to the Member manager service using netcat or a similar TCP client:
   1.1) Run: nc 10.13.37.10 5555

2) When prompted with:
   enter your name:
   type any name (e.g., test) and press Enter.

3) The service will display the main menu similar to:
   Member manager!
   1. add
   2. edit
   3. ban
   4. change name
   5. get gift
   6. exit

4) Type 5 and press Enter to select "get gift".

5) Observe the response:
   your gift:
   1407374883xxxxx
   Member manager!
   1. add
   ...

   - The value under "your gift:" is a large integer that, when converted to hex, appears as a pointer-like address (e.g., 0x7f************).
   - Repeating step 4 within the same session will yield the same value.
   - Closing the connection and repeating steps 1–5 will yield a different value in a new session.

6) To automate reproduction and show both decimal and hexadecimal formats, run the provided Python PoC script (see PoC code) from a machine with network access to 10.13.37.10:5555.
   - The script will:
     - Connect to the service.
     - Supply a name.
     - Select option 5.
     - Parse the leaked integer and print it in decimal and hex.

```
#!/usr/bin/env python3
"""
PoC: Unauthenticated memory address disclosure via "get gift" in Member manager (TCP 5555).

This script:
- Connects to 10.13.37.10:5555
- Sends an arbitrary name at the initial prompt
- Selects menu option 5 ("get gift")
- Extracts and prints the leaked value in decimal and hex
"""

import socket
import re
from typing import Optional


HOST = "10.13.37.10"
PORT = 5555

GIFT_REGEX = re.compile(r"your gift:\s*([0-9]+)")


def recv_all(sock: socket.socket, timeout: float = 1.0) -> str:
    sock.settimeout(timeout)
    chunks = []
    try:
        while True:
            data = sock.recv(4096)
            if not data:
                break
            chunks.append(data)
            # Heuristic: break if we see the menu or the gift line
            if b"Member manager!" in data or b"your gift:" in data:
                break
    except (socket.timeout, OSError):
        pass
    return b"".join(chunks).decode(errors="replace")


def get_gift_value(name: str = "test") -> Optional[int]:
    s = socket.create_connection((HOST, PORT), timeout=5)
    try:
        # Initial banner: "enter your name:"
        _ = recv_all(s)

        # Send name
        s.sendall(name.encode() + b"\n")
        _ = recv_all(s)  # Read menu

        # Choose option 5: get gift
        s.sendall(b"5\n")
        resp = recv_all(s)

        # Optionally read a bit more in case the gift appears in the next chunk
        if "your gift:" not in resp:
            resp += recv_all(s)

        print("Raw response from server:")
        print(resp)

        m = GIFT_REGEX.search(resp)
        if not m:
            return None

        return int(m.group(1))
    finally:
        s.close()


def main() -> None:
    value = get_gift_value()
    if value is None:
        print("[-] Failed to extract gift value. The protocol/output may have changed.")
        return

    print("\n[+] Leaked gift value (decimal):", value)
    print("[+] Leaked gift value (hex):    0x{:x}".format(value))


if __name__ == "__main__":
    main()
```

## Remediation

Remediation should focus on eliminating the memory disclosure and hardening the surrounding memory handling:

1) Remove pointer-based data from client-facing responses:
   - Change the implementation of the "get gift" functionality so that it no longer returns:
     - Raw pointers, or
     - Values derived in a predictable way from pointers or internal addresses.
   - Instead, return:
     - A benign, application-level token (e.g., a random integer generated via a CSPRNG that is not related to memory layout), or
     - A purely cosmetic/static message that does not expose internal state.

2) Introduce strict data abstraction boundaries:
   - Ensure that no user-visible output is constructed directly from internal pointers, addresses, or object references.
   - If addresses must be logged for debugging, log them server-side only, never in responses.

3) Harden memory management and “insecure” paths:
   - Review and refactor the code backing:
     - The "insecure edit" option.
     - Index-based operations (edit, ban).
     - Size-controlled allocations in "add".
   - Enforce comprehensive validation:
     - Bounds-check indices before use; reject negative or out-of-range indices.
     - Impose strict maximum lengths on all user-controlled strings, both for names and usernames.
     - Validate allocation sizes against a safe, documented range; avoid using untrusted sizes directly with malloc/new.
   - Use safer abstractions where possible:
     - Prefer container types with built-in bounds checking over raw arrays.
     - Consider using higher-level languages or libraries that manage memory safely.

4) Defense-in-depth measures:
   - Compile the service with modern mitigations enabled (ASLR, stack canaries, NX, RELRO, PIE).
   - If feasible, run the service with least privilege and apply network isolation controls so that compromise has limited blast radius.
   - Implement logging and monitoring for unusual interaction patterns (e.g., repeated very large inputs, abnormal indices) to detect exploitation attempts.

5) Retest after remediation:
   - After implementing fixes, re-run the PoC and interactive tests:
     - "get gift" should no longer return pointer-like or address-derived data.
     - All menu operations should reject unsafe indices and oversized inputs gracefully without crashes or leaks.
   - Consider adding regression tests that assert:
     - No responses contain raw pointer patterns.
     - Indices and sizes outside expected ranges are handled safely.
